02 · product model

The workspace model

Chat collaborates, canvas hosts live objects, contracts govern action, and marketplace packages install capabilities through versioned, reviewable envelopes.

The corpus explicitly does not claim production marketplace ORPC routes exist yet. Endpoint names are planning contracts until implemented and tested.

03 · package contract

Package and install semantics

One envelope, first-class kinds

  • bundle, app, workflow, skill, command_tool_pack, agent, artifact_template, mcp_addon, provider_integration, and managed_internal stay first-class.
  • Bundles compose package-version refs or embedded seed definitions; they are not wide nullable JSON blobs.
  • Compatibility aliases normalize before persistence/display.

Install behavior varies by kind

  • Installs target immutable packageVersionId, not mutable package ids.
  • Workflow and artifact templates are fork-and-own snapshots.
  • Provider integrations and MCP add-ons enable live connections with runtime credential resolution.

04 · trust boundary

Draft is not preview. Preview is not approval.

The trust model separates user input, renderer state, command preview, trusted approval, and write execution. That separation is the product’s safety claim.

STATE 1
Draft / input

User answers, JSON-render state patches, form fields. Useful, but not authority.

STATE 2
Preview

Non-mutating plan with inputs, effects, host context, and expected receipts.

STATE 3
Trusted approval

Host-authorized approval for a specific command or workflow preview.

STATE 4
Receipt

Evidence that a committed write happened under known package/install context.

Non-negotiable: user text saying “approve,” artifact button clicks, and form submissions are not trusted approval grants. Write/destructive/external commands require preview, host context, trusted approval, and receipts.
05 · decisions

Ratified direction layer

D008–D017 were ratified after a five-lane competitive research pass and internal recon supplement. These are the direction-setting decisions that should shape implementation and UI.

D008
Disambiguate marketplaces. Agent/package marketplace ≠ future commerce marketplace ≠ WhatsApp templates.
D009
Resume amp.pkg v3 trust machinery. Promotion ladder becomes trust tiers; Mockup #5 becomes install screen basis; command registry seeds capability namespace.
D010
Trust tier is visible at discovery. Provenance and runtime permission are different questions.
D011
Fused intent-preview + trust screen. Same execution plan renders the preview and requested permissions.
D012
Per-install kill-switch and activity view. Install-time approval is never permanent trust.
D013
Registered capability-ID namespace. Grants target stable capability ids, not free strings.
D014
Kind-varying install semantics. Templates fork; live integrations enable connections.
D015
Curated review + publisher MFA. Scanning assists; it does not approve third-party publishing.
D016
JSON-first descriptors-only. HTML/custom UI is future exploration only if real needs require it.
D017
Freemium before take-rate. Usage-metric boundaries are the near-term monetization primitive.
