Ratified Decisions D001–D017

The seventeen decisions that fix the marketplace's direction — one envelope, an execution-inert renderer, a trusted-approval write gate, and a registered capability namespace — reconciled against what has actually shipped in code.

Source: docs/product/agent-workspace-marketplace/DECISIONS.md · Corpus 2026-07-07 · Status updated 2026-07-11

How this page reads

This is the reference page: the register is the source of truth for direction, the traceability matrix is the source of truth for current implementation. Where the 2026-07-07 corpus called something "planned" and it has since shipped, the matrix wins and the chip says so. Status vocabulary is fixed:

SHIPPED enforced in committed code + tests today
FIXTURE proven only against a fixture, not the live path
PLANNED direction ratified, no code surface yet
BLOCKED / OPEN started, held on an unresolved decision
MISSING required but absent

Two invariants are not negotiable and appear verbatim across the suite: the JSON renderer is execution-inert (renderer-origin actions never commit trusted writes, D005), and chat is never approval — a write commits only on a host-signed approval carried by trusted host context, with receipts-only success (D006). A model-supplied approval is structurally rejected (model_supplied_approval_is_not_trusted).

Decision register

All seventeen, dense. Each row is the decision in one line and the reason it was made in one line. Contract-baseline rows carry the 2026-07-06 handoff date; research-wave rows carry the 2026-07-07 ratification date.

D001–D017 — decision and rationale, one line each
IDDecisionRationaleRatified
D001 One package envelope, first-class kinds. A single package/version/install envelope keeps bundle, app, workflow, skill, command_tool_pack, agent, artifact_template, mcp_addon, provider_integration, managed_internal as first-class kinds. Amended 2026-07-07: install semantics vary by kind (D014). A bundle installs a whole solution while every component stays independently installable, updatable, and permission-reviewable. 2026-07-06
D002 Install versions, not mutable ids. Installs target an immutable packageVersionId; the mutable package record is a discovery/update pointer only. Enables safe updates, provenance, audit, and copied/forked installs. 2026-07-06
D003 Bundle is default composition, not the only object. bundle is the default composite kind but never replaces first-class app/workflow/skill/tool-pack installs. One-click usefulness must not erase component-level update, fork, permission, or publish flows. 2026-07-06
D004 JSON-render is canonical; HTML is an escape hatch. Deterministic apps author and run as JSON-render manifests; HTML exists only for document/artifact escape hatches, never as the runtime model for command-backed apps. Keeps agent and human state inspectable, patchable, testable, and command-bindable with no sandbox runtime. 2026-07-06
D005 Renderer-origin actions cannot commit trusted writes. Renderer actions may request state updates, previews, approval requests, navigation, and events — never a command commit or trusted approval. Interactive artifacts stay useful behind a hard trust boundary between UI interaction and backend mutation. 2026-07-06
D006 Preview → trusted approval → receipt is mandatory for writes. Write/destructive/external commands and workflow nodes require preview_then_trusted_approval, trusted host context, and receipts. Workflows must act, but writes stay auditable, host-authorized, and reversible by policy. 2026-07-06
D007 The endpoint map is planned, not production API. Marketplace/search/install/workflow endpoint names are planning contracts until implemented as typed ORPC routes. Docs must not imply production support before backend routes, persistence, auth, and tests exist. 2026-07-06
D008 Three "marketplaces" are distinct. Name which is meant: (a) the agent/package marketplace (this corpus), (b) the deferred two-sided commerce marketplace, (c) Meta/WhatsApp message templates (a false friend). The overloaded term produced a phantom "missing PRD" and incompatible data-model assumptions. 2026-07-07
D009 Resume amp.pkg v3 trust machinery. Adopt the §11 promotion-tier ladder as the trust-tier system, Mockup #5 as the install screen, and command-registry.csv capability_ids as the namespace — do not invent parallel concepts. A designed, partially-built internal substrate beats parallel invention and preserves shared vocabulary. 2026-07-07
D010 Trust-tier is first-class and discovery-visible, with published revocation criteria. Every package shows a named tier (first-party / verified-partner / community) at discovery, independent of kind and of runtime grants; demotion criteria are published and falsifiable. Provenance and permission answer different questions; conflating them weakened every surveyed competitor. 2026-07-07
D011 Fused Intent-Preview + Trust screen is the canonical install/commit UX. One screen renders the concrete effect preview from the same execution plan that will run, beside the requested permissions/tier — verbose first use, lighter on repeat. No cosmetic previews, no separate trust modal. The trust gate is the wow moment; same-plan rendering is what makes the preview honest. 2026-07-07
D012 Per-install kill-switch and activity view are first-class. Every install gets a user/org kill-switch (stop trusting forward; no purge guarantee) and a per-install activity view. Install-time approval is never permanent trust. The dominant real-world exploit is post-install; revocation must be as easy as installation. 2026-07-07
D013 Registered, versioned capability-ID namespace before scale. requiredCapabilities and grant targets reference registered capability ids, not free strings. Per-call gating with write-implies-read implication makes namespace and kill-switch one mechanism. Every later control — tiers, grants, audits — keys off capability identity; free strings rot. 2026-07-07
D014 Install semantics vary by kind under one envelope. workflow/artifact_template are fork-and-own snapshots with an immutable packageVersionId; provider_integration/mcp_addon "enable a live connection" with runtime credential resolution and no forked copy. Shareable templates and live provider connections are different problems; one install verb miscasts one of them. 2026-07-07
D015 Curated review + publisher MFA gate third-party publishing; scanning assists, never decides. Human review before listing, mandatory publisher MFA, Zod validation at install and update, re-review on update, async post-listing audit. Scanners (incl. GitNexus cypher gates) pre-screen only. Prompt-injection detection is unsolved industry-wide; the defensible gates are structural, procedural, and credential-hardening. 2026-07-07
D016 §4 posture: JSON-first descriptors-only. Packages declare descriptors mapped to approved design-system components and never ship arbitrary UI code. "JSON first; explore future HTML/custom UI only if the needs require it." The strict posture keeps the execution-inert safety claim absolute and the threat model small, with the door explicitly open rather than ambiguous. 2026-07-07
D017 Freemium before take-rate. A free tier with paid premium capability precedes any take-rate; usage-metric boundaries are the preferred freemium mechanism. Take-rate defers to commerce-marketplace scoping; the cited "Dify 50%" figure is an affiliate rate — do not plan against it. Monetization is unsettled market-wide; reputation-first matches both the research and the May vision. 2026-07-07
