Product Strategy & PRD

The Agent Workspace + Marketplace is a live artifact workspace where an agent creates deterministic, contract-backed mini-apps that install as packages. Contracts and fixtures are stable; the marketplace runtime is deliberately not built until they are.

Source: docs/product/agent-workspace-marketplace/00-product-strategy-prd.md · Corpus 2026-07-07 · Status updated 2026-07-11

Three marketplaces, one overloaded word

Decision D008 splits a term that previously caused a phantom "missing PRD" and incompatible data-model assumptions. Every doc must name which marketplace it means. Only the first is this corpus; the sequencing between the three is unrelated.

Where Sonik is ahead

The PRD's differentiators are invariants, not features — each closes a gap that weakened surveyed competitors. Status here is reconciled against the implementation truth of 2026-07-11, not the corpus's original "planned" language.

Current state against target

The PRD's core tension: the product surface and contract model already exist; runtime endpoints and persistence are held back by policy until contracts stabilize. This is a sequencing choice, not a gap to paper over.

Held back on purpose

Out-of-scope items are deferred because the endpoint docs explicitly label marketplace endpoints as planned typed contracts only. This is the guardrail against reading the PRD as permission to ship writes without the trust chain.

Do not build ahead of the contracts

Deferred: production marketplace ORPC implementation, the visual workflow builder, any new runtime/sandbox for arbitrary HTML apps, and creator monetization / public moderation / external publishing policy. Do not treat this PRD as permission to build writes without preview/approval/receipt, do not expose bundle as the only package kind, and do not store installs against mutable package ids only.
