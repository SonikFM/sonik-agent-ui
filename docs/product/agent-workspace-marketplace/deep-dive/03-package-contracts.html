Package & Bundle Contracts

The marketplace splits every installable thing into three records — a mutable index, an immutable version, and an installation — and lets Zod refuse anything that would blur them. The contracts are shipped and test-locked; the endpoints that will read them are not yet built.

Reconciliation up front

The schemas below live in packages/tool-contracts/src/marketplace.ts and are exercised by tests/unit/marketplace-package-contracts.test.mjs — full chain green. SHIPPED covers the manifest, version, installation, bundle, permission, and command-binding shapes with their superRefine invariants. Everything that would serve these records — ORPC routes, the persistence/RLS tables, the install UI, publish/moderation — is PLANNED. No package has been persisted or installed by any live path yet.

Three records, one immutable target

A package is not one object. MarketplacePackage is a mutable summary/index row that carries a currentVersionId pointer. MarketplacePackageVersion is the immutable install target — its envelope re-checks package identity, semver, schema version, and manifest hash against the manifest it wraps, so a version can never drift from what it claims to be. MarketplaceInstallation records one org/workspace instance and always targets a installedVersionId, never a bare package id.

The invariant that makes the split load-bearing: installs pin to a version, provenance points back to a version. A copied or forked install must carry sourcePackageVersionId, and that source must sit in the same package lineage (installedVersionId and sourcePackageVersionId both start packageId@). Below, the pinned install tracks an immutable version directly; the forked install keeps a line back to the version it branched from.

Install modes × package kinds

Decision D014 fixes four install modes against the ten first-class kinds. pinned tracks a fixed immutable version; copied and forked are editable local copies that must preserve sourcePackageVersionId provenance; subscribed follows upstream but only for low-risk, read-only entries. The only hard, schema-enforced kind bans sit under subscribed (and auto_patch): a bundle may not compose a subscribed command_tool_pack, provider_integration, or managed_internal entry.

Where the ban actually bites

The three subscribed bans are enforced today on bundle composition entries (bundleCompositionItemSchema, message: "Subscribed/auto-patch entries cannot target command packs, provider integrations, or managed internals in v0"). The top-level marketplaceInstallationSchema enforces the read-only-permission requirement for any subscribed/auto_patch install ("Subscribed/auto-patch installations require explicit read-only permissions") but does not itself re-ban those kinds — the read-only rule makes a subscribed write-capable pack structurally impossible regardless. Treat the matrix as the D014 policy; the executable guard is split across those two sites.

Manifest envelope anatomy

Every version wraps one manifest. The field inventory below is the shipped marketplaceManifestSchema shape (defaults shown where the schema supplies one). It is .strict() — unknown keys are rejected outright.

Two invariants not to soften

The manifest is data, never code: findSecretLikePath and findExecutableCodePath walk the whole object and reject secret-shaped keys/values (sk_live_…, bearer tokens, PEM blocks) and code-shaped strings (function(, =>, eval(). And writes are never self-approved: any grant or binding with effect write/destructive/external is structurally forced onto preview_then_trusted_approval with a non-empty requiredHostContext. The renderer stays execution-inert; approval is a host act, never a chat act.
