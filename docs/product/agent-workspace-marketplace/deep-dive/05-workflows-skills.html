Workflows, Agents, Skills & Tool Packs

A workflow is a typed graph, not a prompt. Ten node types compose installable capability, but only two of them can run: a tool_preview the model may call, and a tool_commit the host must sign. Everything between is state the reducer refuses to fake.

Source: docs/product/agent-workspace-marketplace/05-workflows-agents-skills-tool-packs.md · Corpus 2026-07-07 · Status updated 2026-07-12

Node-type reference

The workflow schema enumerates ten node types (workflowNodeTypeSchema, marketplace.ts:205). Enumeration is not interpretation: the shipped reducer (workflow-run-state.ts) and controller (workflow-controller.ts) model five of them with real transitions and gates. The other five parse and validate but the controller returns unsupported_node_type — reserved surface, not live behavior. Only tool_preview is ever exposed to the model; approval and tool_commit stay host-side.

Three invariants the node types encode

The renderer is execution-inert — a tool_preview node cannot carry a mutating effect at all. Chat is never approval — an approve event without hostSigned is structurally refused. Success copy is receipts-only — the committed phase is reached on a semantic-success receipt, never on transport status.

Revised reservation workflow graph

The corpus (2026-07-07) still describes a four-node availability → guest-preview → guest → booking split. The shipped fixture (marketplace-fixtures.ts, Phase 3a revision) collapses the two writes into one compound tool_commit behind a single human Approve, matching how the live reservation path actually runs. There is one reservation_preview (guest + booking, effect none) and one reservation_commit wrapping both writes.

Run state machine

A run is a Zod-validated WorkflowRunState advanced by a pure reducer (Decider pattern) — rejections return { ok: false, reason } with state unchanged, so a bad event can never corrupt a run. The reservation spine runs intake → committed; idle and saving are members of the shared phase vocabulary the spine does not exercise. The one guarded transition is approval_requested → approved: only a host-signed approve crosses it.

Facade pinning

The model does not see the command catalog. Each run pins a facadeToolIds set at start — frozen for the run's lifetime, no toolset churn mid-workflow. The reservation facade is two entries, both preview commands.

Size ceiling
facadeToolIds: z.array(z.string()).max(5) — at most five model-facing tools per run (audit P0). Empty means host-derived; existing definitions parse unchanged.
Preview-only default
The schema's facade superRefine requires every declared facade entry to map to a tool_preview commandId — the only model-callable node type. approval and tool_commit stay host-side click handlers, never model-callable tools.
Reservation facade
["booking.get.availability", "booking.create.booking"] — both preview reads. The write shares booking.create.booking's id but fires only through the host-side commit gate.
Skill-intent short-circuit
A skill node guides; it is not executable authority. With the facade pinned to preview commandIds and writes reachable only through the host-signed tool_commit gate, a runtime skill intent resolves to a preview — it can never become a command directly (command-catalog.ts's draft-only invariant enforces this at the tool layer).

Live pinning is not yet wired

The controller (Phase 3a) is a working interpreter but is not on the live request path — the Phase 3a-2 cutover is an open decision. Grant synthesis reproduces the live policy through the capability registry, but pinning is BLOCKED on registry coverage: 8 registered capabilities against 72 live catalog commands.
