System Map & Boundaries

Four layers, one trust boundary. The renderer and the agent live on the untrusted side and may only ask; writes cross the boundary only behind a host-signed grant. The map below is the authority diagram, not an org chart.

Source: docs/product/agent-workspace-marketplace/02-system-map-and-boundaries.md · Corpus 2026-07-07 · Status updated 2026-07-11

The trust boundary

The corpus system map is a flat mermaid graph; the load-bearing fact it under-draws is which edges cross the trust boundary. Everything left of the line runs inside the agent-UI iframe and is model-influenced — the Agent Chat proposes and the Canvas renders, but neither can authorize an effect. Everything right of the line is the trusted host runtime, where a real operator and a host signature are the only things that unlock a write. @sonik-agent-ui/tool-contracts is the shared law underneath both sides; it is a type/schema substrate, not a runtime crossing.

  • host-signed authority / trusted edge
  • gated capability + commit gate
  • contracts shared type substrate
  • not built planned, no runtime

Host-context donation & signed-grant path

A write is not a message the model sends — it is a grant the host signs. The path below is how authority actually crosses the boundary, reconciled to the shipped one-Approve reservation behavior. The model never holds the pen: model_supplied_approval_is_not_trusted is enforced structurally in workflow-run-state.ts, so a model-fabricated approval object is rejected before it reaches the gate.

1 · Host mints envelope identity, org, approved capability ids packed into the host-context envelope
→
2 · Donate to iframe envelope crosses into the agent UI as authority-bearing context, not display metadata
→
3 · Renderer builds affordance one shared approval-affordance builder (D011 seam) feeds intake and reservation, byte-identical
→
4 · Operator approves, host signs approval happens in host chrome; host signs the grant — chat cannot substitute
→
5 · Capability gate evaluateCapabilityAccess checks the signed grant; write-implies-read, kill-switch via revokedCapabilityIds
→
6 · Commit + receipt phase reaches committed only on a semantic-success receipt, never on the request

The A2 reservation commit path is merged and live: a preview tool, a host-signed approval card, and /api/reservation/commit. The 2026-07-08 handoff's "A2 UI half open" note is stale. The reservation workflow fixture was revised to one compound commit node so the graph matches the shipped single-Approve behavior.

Non-negotiable invariants

Do not soften these

No UI or artifact layer may bypass a contract-layer write gate. Runtime endpoints must not redefine package semantics outside @sonik-agent-ui/tool-contracts. Host context is an authority boundary, not display-only metadata. The renderer is execution-inert, chat is never approval, and success copy is receipts-only.

Everything on this page reduces to one rule: authority lives on the trusted side of the boundary and is spent only through a host-signed grant. The contracts are the source of truth today; the runtime that consumes them is still, mostly, a set of typed promises.
