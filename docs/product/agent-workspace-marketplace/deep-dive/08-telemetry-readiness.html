What must be proven before Agent Workspace + Marketplace ships — and the ledger of which proofs already hold. Unit coverage is real and green; live endpoint telemetry is still a gap.

The readiness-gate ladder

Readiness is not one test suite; it is four gates a change must clear in order. The bottom two rungs are backed by shipped, full-chain-green contracts. The top two — anything that needs a live request path or a running browser — are not yet built. Live coverage stops where the reducer stops.

The evidence pipeline

A release claim is only as good as the artifact behind it. The invariant is one-directional: user-facing success copy derives from a semantic-success receipt, never from transport status or a chat transcript. The reducer already enforces the last hop; the middle hops (live event → Pipe-B → receipt) are the telemetry gap.

Telemetry event install / run / update / approval fires with packageVersionId, installationId, previewId, approvalId.
→
Pipe-B traceable event sink; joins the chain on workflowRunId. No marketplace stream yet.
→
Receipt semanticStatus: "success" | "failure" with receiptRef. Phase committed only on success.
→
User copy "Done" is rendered from the receipt, never from HTTP 200.

Green nodes are enforced in shipped code; amber nodes are planned wiring.

Receipts-only success copy SHIPPED

The reducer refuses to enter committed without a semantic-success receipt for every commit node — "success copy derives from receipts, never transport status" (workflow-run-state.ts:12-13, 290). A friendly label can never outrun the receipt behind it.

Live join key PLANNED

workflowRunId (the reducer's runId) is the intended join key across event, Pipe-B, and receipt. It exists in the shipped state machine but is not yet emitted on a live telemetry stream.

Telemetry event vocabulary

The planned event surface for marketplace actions. Every event carries immutable ids so a run can be reconstructed after the fact; workflowRunId is the join key that stitches a preview to its approval to its receipt. None of these emit on a live stream yet — the contracts that define the ids are shipped; the endpoints that fire the events are not.

FIXTURE marks event shapes the shipped reducer and its templates already model in tests; MISSING marks events with no producer at all. The A2 reservation commit path (/api/reservation/commit) is the one live commit surface today and is SHIPPED, but it does not yet emit these named marketplace events.

Accessibility requirements ledger

Accessibility and agent-readability are both first-class readiness gates — a custom approval widget that a keyboard or screen reader cannot drive is not shippable, and neither is one an agent cannot read. These requirements are product law today; the component-level tests that prove them wait on the UI itself.

Keyboard operability
Every approval button, package card, and command-backed form control is reachable and operable by keyboard alone. MISSING — needs component tests post-UI.
Focus management
Focus is visible (token :focus-visible outline) and moves predictably through preview → approval → receipt. MISSING
Screen-reader labels
Buttons, cards, and flows carry readable labels; no icon-only unlabeled controls on approval paths. MISSING
No inaccessible custom widgets
Custom widgets for approvals or command-backed app forms must not ship if inaccessible. OPEN — binding rule; enforcement pending UI.
Agent-readable state
Install pages expose machine-readable getPageContext() with package/version/install ids; semantic actions include disabled reasons. PLANNED
Receipts stay visible
Friendly UX labels must not remove technical receipts or host-context requirements. SHIPPED — enforced at the reducer.

What developers must not do

Do not mark marketplace workflows production-ready from unit tests alone; do not hide receipts or host-context requirements for simpler UX; do not ship inaccessible custom widgets for approvals or command-backed forms. Passing pnpm test proves broad current unit coverage — it does not prove live endpoint telemetry.

Non-negotiable invariants

  • Release claims require evidence, not chat-transcript confidence — chat is never approval.
  • Success copy derives from receipts, never transport status. The renderer is execution-inert.
  • Friendly UX labels must not remove technical receipts.
  • Accessibility and agent-readability are both first-class readiness gates, not follow-on polish.
