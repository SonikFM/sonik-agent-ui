Permissions, Approval & Trust Boundary

The line between what a model can say and what the host will do. Chat text is never approval; a write side effect fires only behind a host-signed gate — and as of this week that gate is enforced by the reducer's structure, not by reviewer discipline.

Source: docs/product/agent-workspace-marketplace/07-permissions-approval-trust-boundary.md · Corpus 2026-07-07 · Status updated 2026-07-12

The approval sequence

A run walks one phase pipeline. The first three phases are drivable by untrusted inputs — model output, chat, JSON-render patches, artifact state. Everything from approved onward requires a host-signed marker the model cannot forge. The boundary is not a convention between those two halves; it is where the reducer refuses transitions.

→
preview_readyNon-mutating command preview: commandId, stableInputHash, effect, approvalRequired. No write runs.
→
approval_requestedPreview handed to the host surface. approvalState.status = "requested".
→
approvedHost-signed only. hostSigned: true or the schema refuses to represent it.
→
committingInjected write callback invoked — never before this gate passes.
→
committedTerminal only on a semantic-success receipt, once every commit node is done.

Phases per WORKFLOW_RUN_PHASES in packages/tool-contracts/src/workflow-run-state.ts. Off-path terminals error and cancelled omitted from the strip.

Three separate states

The corpus names three states that the contracts keep disjoint. Draft and preview are cheap and model-drivable; only the third carries authority. Conflating any two is the class of bug this boundary exists to prevent.

Draft, preview, and trusted-approval states, with what each may and may not carry
StateWhat it isMay carryMust never carry
Draft / input User answers, JSON-render state patches, form fields. Arbitrary user text, including the literal word "approve". Trusted principal credentials or approval tokens.
Preview Non-mutating command/workflow preview: inputs, effects, required context, expected receipts. Human-readable command shape and stableInputHash. Any mutating side effect — preview runs no write.
Trusted approval Host-authorized approval for a specific preview. hostSigned: true and the approvedCommandIds it authorizes. Model provenance — the schema refuses approved without the host-signed marker.
Invariants — not softened

  • Write / destructive / external effects require host context and a commit gate. SHIPPED

Why two independent commit gates

The controller re-checks approvalState.status === "approved" && hostSigned before it ever calls the injected write callback, on top of the reducer's own commit_started refusal. A write side effect can therefore never fire on an unapproved run even if a future caller skips the reducer transition entirely. Belt and suspenders, on purpose — the mutation is the irreversible act.

Three-layer grant resolution

One effective mode is pinned per registered capability at run start and frozen for the run's lifetime, so two enforcement paths can never disagree mid-run. Three authority inputs feed it, each with deliberately different absence semantics; the pin is most-restrictive-wins across all three.

Direct grants precede implication

Within the capability-grant layer, a write grant covers its declared reads (booking.create.booking implies booking.get.availability). But implication exists only to expand sparse grants, never to narrow a capability that carries its own grant: if a capability has a direct grant, only direct grants resolve it. Otherwise a restrictive grant on X would bleed onto everything X implies.

Default-deny
Capability grants: no grant, unregistered id, revoked descriptor, or kill-switched id all evaluate to off.
Direct beats implied
Matching direct grants form the resolution pool; implied grants are used only when the pool is empty (evaluateCapabilityAccess).
No privilege escalation
An implies edge may never point at a more dangerous effect — implication can only lower privilege (write implies read), enforced at registry parse.
Frozen per run
The controller resolves the pin once at run start and freezes it; facadeToolIds and pinned modes never change mid-run.

Kill-switch mechanics

Revocation and the capability namespace are one mechanism, not two. There is no separate revocation machinery: revoking a capability id makes the next call evaluate to off. Because a revoked granting capability contributes nothing to resolution, its implications stop flowing in the same step.

Revocation = off next call
A kill-switched id (revokedCapabilityIds) returns kill_switch_revoked on the next evaluateCapabilityAccess; a descriptor marked status: "revoked" returns capability_revoked. No in-flight commit is retroactively undone — the gate simply bites at the next call.
Implication flow stops
A revoked granting capability is skipped before its implied ids are expanded, so revoking a write capability also drops the reads it would have covered — unless those reads carry their own grant.
Default-deny reasons
capability_not_registered, capability_revoked, kill_switch_revoked, and no_grant all pin to off; only granted yields a mode.
Pins to off
In the three-layer pin, a revoked id forces off regardless of what toolPermissionModes or approvedCommandIds say — most-restrictive-wins guarantees it.

Per-install kill-switch is D012; the namespace and gating are D013. Both in packages/tool-contracts/src/capability-registry.ts. The revocation UI is PLANNED — the enforcement mechanism is live, the operator surface is not.

Why per-call, not install-time-only

The research corpus converged from three lanes on one finding: install-time vetting is where every 2025–2026 marketplace breach succeeded, and it is insufficient on its own — the worst exploits are runtime and update-time, against an already-approved, already-installed capability. Our gate re-evaluates on the next call, so a trusted-publisher malicious update or a runtime injection meets a live gate, not a frozen attestation.
