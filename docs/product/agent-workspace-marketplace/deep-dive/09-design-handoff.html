Three-zone workspace model

The design north star is one durable workspace split into three zones: a chat / steps rail on the left, the canvas artifact in the center, and a collapsible inspector on the right, under a header that carries subject chips, mode, share, and logs. The zones are not cosmetic panels — each one owns a distinct half of the trust boundary. The chat rail collects intent and answers but never approves. The canvas renders the live artifact but is an execution-inert renderer: it draws state, it does not mutate it. The inspector is where provenance, receipts, and version history stay first-class rather than buried.

The mutation path threads through all three zones in a fixed order — answers are collected in chat, the manifest is validated and previewed on the canvas, and only an explicit approval hands off to the trusted host, which returns a receipt to the inspector. The renderer never shortcuts a step.

Design-constraints ledger

Two ledgers gate every screen. The first is Dan's binding taste rules — hard nos that reject a design regardless of merit. The second is the set of engineering invariants the UI cannot render away, drawn from the handoff's non-negotiables.

Taste rules (binding)

Gradients
Banned everywhere, including gradient text and backgrounds. Flat surfaces from the token ramp only.
Emoji
No emoji as UI structure or iconography unless brand-approved.
Left/right stripe cards
No rounded card with a colored side-stripe accent. Callouts use a full border and tinted surface.
Inter / Roboto
Banned. Type comes from the token stack (--font-sans, --font-mono) only.
Ethos
“One thousand no’s for every yes.” Decoration is a cost, not a default.

These four bans govern the product being designed and this deep-dive suite that documents it — the ledger describes the page it is printed on.

Engineering invariants (cannot be designed away)

Version visibility
Design cannot remove packageVersionId / version from install and update detail views.
Permission semantics
Controls must expose off / ask / allow wherever applicable.
Answer ≠ approval
QuestionCard surfaces collect typed input only; they must not execute tools or mutate Booking/Amplify.
Approval boundary
The approval card must map to a trusted host session approval; a model-supplied approval is not trusted.
Renderer is inert
JSON-render / ask-user surfaces are presentation and input collectors; execution stays in trusted host/controller tools.
Receipts on mutation
Every mutation returns a receipt / correlation id. Success copy is receipts-only.
Machine-readable context
Page context via window.__sonikAgentUI; host embed via window.__sonikAgentHost, not coordinate clicks.
Provenance first-class
Provider, fetched_at, license class, receipt, and source confidence stay visible; do not bury them behind decorative cards.

D011 — fused Intent-Preview+Trust screen

D011 is the ratified decision that install and commit approvals use one fused screen: the concrete command/effect preview rendered from the same execution plan that will run, shown alongside the requested permissions and trust tier. Verbose on first use per category, lighter on repeat. No cosmetic previews; no separate blocking trust modal. The shared approval-affordance builder (Phase 2, shipped) is this decision expressed in code.
