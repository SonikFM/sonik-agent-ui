Opportunity space

The program exists to let operators build command-backed business apps from chat, install them safely, and reuse them as versioned packages — without ever confusing a UI click or a chat "approve" with authorization. The direction below is outcome-first; each opportunity carries a concrete proof in the contracts or tests, not a feature wish.

Release slices R0–R6

The corpus defined a seven-slice ramp from contract docs to a published marketplace. R0 shipped as the doc set itself. Since the 2026-07-07 corpus, the runtime spine advanced further than the roadmap predicted: an R2-adjacent workflow state machine and generic controller landed as green, committed libraries, and the R5 reservation-commit path went live. Neither is the same as a wired marketplace — the timeline marks where real code now sits versus where the plan still points.

R4 gating is blocked on registry coverage

Grant synthesis (Phase 4) reproduced the live policy exactly through the registry, but live pinning is BLOCKED: only 8 capabilities are registered against 72 live catalog commands. Until the registry covers the catalog, R4's per-command install permissions cannot pin against real grants. The kill-switch (revokedCapabilityIds) and write-implies-read implication are in place; coverage is the open decision.

Scope ledger

What the program commits to preserve, and what it explicitly refuses to build yet. The out-of-scope column is not backlog — it is a set of guardrails: shipping any of these early would break the preview → approval → receipt contract or make DOM text the source of truth.

In scope — invariants every slice preserves

  • Preview → trusted approval → receipt semantics for every write, on every slice.
  • R2 exposes machine-readable workflow state without granting command authority — the renderer stays execution-inert.
  • R3 / R4 install and update against an immutable packageVersionId.
  • Approval is host-signed and structurally required; chat "approve" text is never authorization.
  • A phase is committed only on a semantic-success receipt — receipts-only success copy.

Out of scope — deferred, deliberately

  • Production data model / migrations for packages or installations.
  • Design-approved marketplace install screen.
  • Public publish / moderation policy; trust-tier field; revocation UI.
  • A visual workflow builder before package / install / update contracts are stable.
  • Any jump from R0 straight to live mutating execution without R2 / R3 / R4 proof.
Direction ratified against the July 2026 ecosystem scan

The stateful-runtimes landscape review (docs/research/stateful-runtimes-landscape-2026-07.md) confirmed the spine: keep the Decider-pattern reducer; no OSS engine combines JSON workflow documents, a single interpreter, and a typed host-signed approval boundary. The slices above build on that seam rather than adopting an external engine.
