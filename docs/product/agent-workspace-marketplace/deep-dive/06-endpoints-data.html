Proposed data model

Seven tables mirror the contract boundaries. Lineage runs left to right — package → version → installation — and every mutating path terminates at a receipt only after a host-signed approval. Three tables already have committed Zod shapes in packages/tool-contracts/src/marketplace.ts; the rest are proposed with no schema yet.

Persistence notes

None of these tables is committed to a runtime yet, which is deliberate — the July 2026 stateful-runtimes scan (docs/research/stateful-runtimes-landscape-2026-07.md) settled the direction before the schema hardens. Keep the Decider-pattern reducer as the core; persist through Durable Objects when durability lands.

Storage-shape constraints carried forward from the July scan
Durable Objects
Read-your-write consistency — the right home for approval and run state, where a request must observe its own prior write. workflow_approval_requests and workflow_run_previews need this; the reducer stateSchema is structurally close to a DO's persisted state.
KV
Eventually consistent — a documented footgun for state that gates execution. Acceptable for discovery-side reads (marketplace_packages index) where staleness is tolerable, unsafe for anything on the approval path.
D1
Hard 10 GB cap per database. Immutable, ever-growing tables — marketplace_package_versions, marketplace_installation_events, command_receipts — must plan for that ceiling before they are the system of record.

Ratified: the typed host-signed approval boundary remains uncommoditized — no OSS engine combines JSON workflow documents, one interpreter, and a typed host-signed approval seam. It is the moat, and it is what workflow_approval_requests exists to persist.
