Command-Backed Apps & JSON-Render

Stateful mini-apps without a new runtime engine: JSON-render is the canonical app definition, command bindings add preview/commit seams, and trusted host approval stays external to renderer state. The renderer is execution-inert by contract.

Source: docs/product/agent-workspace-marketplace/04-command-backed-apps-json-render.md · Corpus 2026-07-07 · Status updated 2026-07-11

Renderer action vocabulary

A command-backed app is a CommandBackedAppDefinition (packages/tool-contracts/src/marketplace.ts:158-203): identity, a jsonRender block, a list of actions, a list of commandBindings, and required host context. Every control the app exposes is an action whose kind is drawn from a fixed enum. The enum lists six values; the schema's superRefine rejects one of them on any renderer action, so five are usable. The rule is deliberately asymmetric — the renderer may reference state, preview, approval, navigation, and event semantics, but it may never become a write executor.

Action kind enum from marketplaceAppActionSchema (marketplace.ts:134-140), gated by the app-definition superRefine (marketplace.ts:181-200).
action kind Status What it may cause Binding rule enforced by schema
state_update ALLOWED Patch local renderer state at the paths in writesTo. No backend contact. None — pure client state.
command_preview_request ALLOWED Ask the host to compute a read-only preview for a bound command. If bindingId set, it must resolve and its mode must be preview (marketplace.ts:185-191).
approval_request ALLOWED Surface the host approval affordance for a commit binding. It is a request, never a grant. If bindingId set, it must resolve and its mode must be commit (marketplace.ts:193-199).
navigate ALLOWED Move within the app/workspace surface. No effect leaves the client. None.
emit_event ALLOWED Emit a structured event onto the agent/host bus. Signal only — carries no write authority. None.
command_binding_ref SCHEMA-REJECTED Would let a renderer action point straight at an executable binding — a write button. Always rejected: "Renderer actions must request preview/approval, not bind directly to executable command bindings" (marketplace.ts:181-183).

The value stays in the enum so a definition that reaches for it fails validation with a specific message rather than being silently dropped. Preview → trusted approval → commit

Write effects reach the backend on exactly one path. A renderer command_preview_request asks the host to compute a preview; an approval_request surfaces the host approval card; the human approves in the host, which mints a host-signed grant; only then does the matching commit binding run. The two renderer steps are inert — the trust boundary sits between approval_request (renderer, inert) and the host-signed grant (external to renderer state).

Execution-inertness ledger

The invariant is that a JSON-render app is inert: its state is not a backend write, and its approval requests are not grants. This ledger names, effect by effect, what an installed app copy can and cannot cause — and what stops it.

A JSON app CAN

  • Mutate its own jsonRender state via state_update — draft edits, form fields, view toggles.
  • Request a read-only preview from the host through a preview binding.
  • Surface the host approval card via approval_request.
  • Navigate its own surface and emit_event signals to the agent/host bus.
  • Expose an HTML presentation escape hatch — iframe-sandboxed, over a canonical JSON state ref (htmlPresentationEscapeHatchSchema, marketplace.ts:143-156).

A JSON app CANNOT

  • Execute a write or commitCommand from a button click — command_binding_ref actions are rejected outright.
  • Mint or forge trusted approval — grants are host-signed and external to renderer state.
  • Reach a commit binding without a same-command preview and a host-signed grant.
  • Store secrets or trusted-approval state inside artifact state.
  • Run write/destructive/external effects without preview_then_trusted_approval and a non-empty requiredHostContext.

Attempted effect Reaches backend write? Stopped / gated by

Commit binding, preview + host-signed grant + host context YES — the one legal path preview_then_trusted_approval satisfied; receipt on semantic success
Invariants (do not soften)

Renderer state updates are not backend writes. Approval requests are not trusted approval grants. Commit bindings require preview_then_trusted_approval and requiredHostContext for write effects. Chat is never approval; success copy is receipts-only.

What developers must not do

Do not make artifact button clicks execute commitCommand directly. Do not store secrets or trusted-approval state in JSON-render artifact state. Do not use HTML as the default command-backed app runtime unless a sandbox/runtime contract is later approved.
