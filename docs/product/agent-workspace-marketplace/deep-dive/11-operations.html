Operations Runbook

How the marketplace corpus and its shipped runtime primitives are kept honest: the verification gate every change runs, the incident paths for a suspect package, a stuck workflow, or a pulled capability, and the invariants that are not negotiable under pressure.

Operating posture

This runbook governs a corpus whose contracts have partly shipped and whose product surface has not. The capability registry, workflow-run state machine, and the reservation commit path are live at the contract layer; the marketplace routes, persistence, and every operator UI (install, moderation, revocation) are still on paper. The operational discipline follows from that split: prove the contract, do not assume the product.

Do not work from memory

The corpus is a contract, not a chat summary. Stale docs are blockers for implementation PRs touching the same contract area, and unknown implementation status is called out as a gap — never implied as done. Verify cited source lines still exist before you edit.

The verification gate

Every change to a cited contract runs the standard sequence before it is claimed done. Each step is a process that exits non-zero on failure; a non-zero exit is the gate, not a log line to skim. The sequence is the runbook's Prove it block verbatim.

Incident decision paths

Three recurring incidents, each expressed as the actual mechanics that exist today. A gated node is a decision point; the terminal node is the resolution. None of these paths depends on an operator UI — they run through the contract layer and the verification gate.

Kill-switch procedure

The revocation mechanism ships at the contract layer through the capability-ID registry; the operator surface does not. Enforcement is deny-on-next-call — it does not interrupt a call already in flight, and it only covers capabilities that are registered.

Operational invariants

These hold under deadline and disagreement. They are not style preferences; violating one is the incident.

Non-negotiable invariants and where they are enforced
InvariantWhat it means operationallyEnforcementStatus
No deploy without an explicit request Never deploy a service — booking-service included — off a stale branch or on inference. A green build is not a deploy authorization. Human decision; deploy is out of the automated gate ENFORCED
Sentinel-exit-code gating Each proof step exits non-zero on failure; the merge is gated on exit codes, not on reading output. A skipped step is a failed gate. Standard verification sequence (five steps) SHIPPED
Evidence, not assertion A built/planned claim is backed by a passing proof and reconciled docs. Endpoint implementation is never claimed from an endpoint planning doc. Doc-update rule + Prove it block ENFORCED
Docs are a contract Any PR changing a cited surface updates this corpus or states why it stays valid. Stale docs block PRs in the same area. Review gate; CI wiring for prove-it commands not yet built OPEN · CI
Chat is never approval Commit phase requires a host-signed approval and a semantic-success receipt; a model-supplied approval is structurally rejected. Workflow-run state machine reducer SHIPPED
