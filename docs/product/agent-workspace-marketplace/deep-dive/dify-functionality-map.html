Studio — app creation

Where a Dify build begins: an entry choice, a small set of app archetypes, and a curated template catalog. The archetypes map cleanly onto our package kinds; the catalog taxonomy does not exist yet.

How Sonik meets this: Sonik's package kinds are the archetypes — an organizer creating something picks a workflow, an app, or an agent, each authored as a JSON-first render document (D016) and installed under a pinned version (D014). The organizer never faces a blank canvas; they answer a short setup. The multi-app template catalog and the standalone sidebar are simply unbuilt — they arrive with the marketplace routes; nothing here is a deliberate skip.

Chatbot Orchestrate surface — no-canvas app config

How Sonik meets this: The two load-bearing pieces already ship — the organizer debugs in a live preview and picks a Gateway-discovered model, exactly as here. The inline-insert editor tricks, endpoint-bound variables, and vision are unbuilt; knowledge is in scope but deferred to its own unpack (see Knowledge).

Workflow / Chatflow canvas — the visual builder

The React-Flow graph editor. We model the workflow document and its execution semantics; the visual editor chrome around it is not built. The Human Input node is where their model meets our approval boundary.

How Sonik meets this: We model the workflow document and its execution, not the visual editor — the canvas is a debug-only P3 surface and organizers never see it (workflows are invisible to them by design). Human Input is where their graph meets our host-signed approval boundary, which ships today: resuming a write demands an approval the host signed, and the organizer approves it on the D011 Review card. Control and terminal nodes map to the schema; compute nodes, the editor chrome, ENV, moderation, and chat feature toggles are unbuilt.

Knowledge — in scope, deferred to its own deep-dive

Ratified in-scope (Dan 07-13): "if we make knowledge incredibly simple and effective, we will cook." Dify's stack proves the ceiling; ours must hide all of it. Structural note kept: their custom pipeline reuses the same workflow-as-data pattern.

How Sonik meets this: In scope, deliberately deferred to its own unpack session (Dan 07-13) — so these rows read "in scope · unmodeled," not unmapped: the requirement is ratified, only the mechanism is a later design pass. The proposed shape is the "What your agent knows" panel below: the organizer's live pages are the knowledge, dropped files just work, and every word of retrieval vocabulary stays hidden. The win we're chasing is that the ingestion, chunking, and freshness all run invisibly.

Proposed UX — "What your agent knows"

One panel, three sources, zero retrieval vocabulary. (1) Your pages are the knowledge: the agent reads the organizer's live Sonik surfaces — event pages, policies, menus, lineups — through the sonik-agent-ui contract. Nothing to upload, never stale, and an answer cites the page it came from. (2) Drop anything else in: PDFs and docs just work; chunking, embeddings, retrieval settings, and metadata filtering are never shown — the pipeline runs invisibly. (3) Learned service knowledge: customer-service best-practice datasets that self-organize and garbage-collect — when a source page changes, facts derived from it expire instead of lingering as wrong answers. Plus a "teach it" box: type a correction once, it becomes a pinned fact with provenance. What the organizer sees is a source list with freshness stamps and per-source on/off — that is the entire surface. Deep design reserved for a dedicated unpack session.

Integrations — the workspace-level plugin system

The plugin taxonomy that our package kinds mirror — and where the sharpest contrasts live. Swagger-API-as-Tool is a confirmed convergence with /sonik-tool-creation; their permission model and ambient credential are exactly what we deliberately go finer than.

How Sonik meets this: Their plugin taxonomy is our package kinds, one-to-one; install sources and pinned-by-default updates map to D014. The sharp edges are deliberate: their single ambient credential and their two-question permission model both become per-call capability gating through capability-registry.ts, which ships today. The Swagger-paste convergence is real as a generation pattern — but we generate the catalog internally from the booking manifest, so an organizer never pastes an OpenAPI doc.

Confirmed convergence — Swagger-API-as-Tool → /sonik-tool-creation

Paste an OpenAPI/Swagger doc and Dify generates a full toolset instantly — witnessed: the Sonik booking OpenAPI produced 113 tools in one paste. That is precisely the /sonik-tool-creation pattern, and their UX proves it. What we add on top: effect-typing per operation, per-call gating through the capability-registry, and an approval binding to the host-signed commit. Their auth is one ambient credential (an API key in a header or cookie) applied to all generated operations with no per-op gating — that gap is the whole reason our version exists. One difference in framing: our 113 commands are generated internally from bookingOperationManifest and drift-verified, never pasted by an end user.

Proposed UX — Channels, not trigger plugins

The organizer never sees "webhook trigger plugins." They see a Channels panel on their agent: WhatsApp, SMS, the web widget — voice later. "Connect WhatsApp" is one flow: provision a phone number right there (numbers are a purchasable Sonik resource), pick which agent answers it, set hours and the escalation policy. Each channel gets one in-service toggle — Dify's own per-surface switch, aimed at channels, because half the value is meeting guests where they already are. The endgame primitive: an agent can own its number — personal or customer-facing — so it sits in a guest's contacts like a staff member; "hire staff, not deploy software" made literal. Strategies never appear as a concept the organizer touches: bundles ship behavior preconfigured.
