Sonik Agent runtime architecture / 2026.07

Architecture research synthesis

One control plane. Multiple execution depths.

Sonik should not become another coding harness or workflow engine. It should become the governed workspace that authenticates context, routes work to the correct runtime, preserves every consequential event, and lets people supervise the same agent from web, terminal, voice, or messaging.

RESEARCHED 2026-07-16 SONIK AGENT UI ARCHITECTURE READINESS OFFICIAL + REPOSITORY SOURCES
The recommendation

Keep Sonik’s workflow journal, approvals, host authority, and telemetry as the source of truth. Prove the full coding loop with AI SDK Harness plus a full sandbox. Evaluate AgentOS as a lightweight execution lane—not as a replacement platform.

01 / Decision

The short answer

The architecture is substantially closer than a greenfield six-to-eight-week platform build. Sonik already owns many of the difficult control-plane contracts. The missing work is a bounded execution plane, a truthful capability boundary, and a coherent supervisory experience.

✓

Adopt a two-depth execution model

Start with a full Linux sandbox for the end-to-end vertical slice. Add AgentOS only after its faster lane demonstrates real value for repository analysis, transforms, bindings, and lightweight commands. Both lanes emit Sonik’s normalized run events and remain subordinate to Sonik authorization.

03 / Foundation

What Sonik already owns

The repository already contains the bones of a serious workflow control plane. Replacing these with another framework would add migration risk and divide authority across multiple sources of truth.

Existing hard contracts

  • Workflow nodes, bindings, waitpoints, budgets, and leases
  • Replay, approval, idempotency, and external-effect claims
  • Lease-fenced compare-and-swap run driver
  • Durable workflow and run-event storage
  • Text, reasoning, tool, artifact, error, and usage events
  • Workflow, trace, run, and session telemetry correlation
  • Reusable workspace, chat, canvas, and devtools surfaces

Missing execution seams

  • Concrete workflow executors beyond current stubs
  • Sandbox lifecycle and attachment contract
  • PTY, process, file, diff, preview, and network events
  • Repository and sitemap identity in the run context
  • Resumable live SSE or WebSocket transport
  • Provider capability and readiness reporting
  • Delayed retry and queue conformance

05 / AgentOS

What AgentOS changes

AgentOS makes a two-depth execution plane credible. It is not a complete replacement for a Linux sandbox, but its virtualized kernel, bindings, sessions, network controls, and multiplayer events could make routine agent work dramatically lighter.

Why it matters

  • In-process or sidecar virtual machine architecture
  • Virtual filesystem, processes, pipes, PTYs, and sockets
  • Deny-by-default mounts, network, and host process access
  • Long-lived, streamed, replayable agent sessions
  • Host functions exposed as typed CLI bindings
  • Broadcast events for observers and copilots
  • Signed preview URLs and full-sandbox escalation

Why it cannot be the only lane

  • No arbitrary apt, yum, or binary installation
  • No Docker, kernel modules, eBPF, GPU, or hardware access
  • No inotify or filesystem watching
  • Running processes and shells do not survive VM sleep
  • Codex adapter is beta
  • Security model is still undergoing review
  • Host bindings carry full authority if mis-scoped

AgentOS adoption rule

Prototype the smallest core or sidecar integration and translate its session events into Sonik run events. Do not adopt Rivet workflow or actor state as a second authoritative workflow ledger. Host bindings must pass the same Sonik grant, approval, idempotency, and external-effect checks as every other command.

Benchmark interpretation

AgentOS publishes compelling start-time and density claims. They are vendor-supplied and directional. Sonik should evaluate them using its own repository sizes, command mix, concurrency, model latency, and audit requirements before making a product promise.

Codex-specific reality

AI SDK’s current Codex harness normalizes shell and web search, translates file patches into file-change events, and relays custom host tools through a CLI bridge. Native Codex tool approvals and arbitrary MCP execution do not map cleanly into the AI SDK abstraction yet.

AgentOS opportunity

AgentOS bindings can expose typed host capabilities as discoverable CLI commands. Used behind Sonik’s authority gateway, this can produce a concrete answer to “what can the agent actually call right now?”

08 / Sequence

Divide and conquer

Independent lanes can start after the event and authority contracts are frozen. Avoid a broad provider abstraction until two implementations have proven the actual seam.

TRAIN 0

Freeze contracts

Define repository manifest, execution attachment, normalized process/file/preview events, capability readiness, and resumable cursor.

Gate: fixtures replay without a live provider.
TRAIN 1

Prove the full sandbox

Attach a repository, edit a file, run a targeted test, start the hot server, and return a signed preview through AI SDK Harness.

Gate: one end-to-end run survives client reconnection.
TRAIN 2

Build the Live Workbench

Reuse workspace, chat, canvas, and devtools primitives. Add activity, files, diff, preview, verification, approval, and readiness views.

Gate: state is clear without raw logs.
TRAIN 3

Evaluate AgentOS

Test bindings, repository transforms, concurrency, suspend behavior, security boundaries, and event translation.

Gate: measurable advantage with no authority or replay loss.
TRAIN 4

Add the TUI

Connect AI SDK TUI to Sonik’s existing transport rather than creating a second backend or history.

Gate: web and TUI supervise the same run.
TRAIN 5

Extend to edge and voice

Add Cloudflare when global topology is needed and connect Pipecat transports to the shared session contract.

Gate: new channels add no new authority model.

Do not build

  • A second workflow engine beside Sonik’s run journal
  • A generic provider framework before a second provider proves it
  • A terminal as the primary product experience
  • A global tool catalog presented as current availability
  • Provider-specific events leaking directly into the UI
  • Secrets or host environment values copied into guest runtimes
  • Model-native approval treated as organizational authority
  • Separate histories for web, terminal, voice, or WhatsApp

Strategic conclusion

The edge is not a faster terminal or another agent loop. It is a governed, host-aware workspace that places each task in the cheapest capable runtime while preserving one truthful history of intent, authority, execution, and proof.
