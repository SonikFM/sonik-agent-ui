## Memory infrastructure

### Entity model
Core value types live in `src/openhuman/memory/traits.rs`, now **re-exported verbatim from the `tinycortex` crate** (migration decision: one source of truth, 30+ call sites keep old import paths): `MemoryEntry`, `MemoryCategory`, `MemoryTaint`, `RecallOpts`, `NamespaceSummary`. The `Memory` trait (`store`, `store_with_taint`, `recall`, `recall_relevant_by_vector`, `get`, `list`, `forget`, `namespace_summaries`, `count`, health check) is the storage-backend abstraction; `UnifiedMemory` is the concrete SQLite implementation.

**`MemoryTaint` is the standout pattern**: a two-variant provenance enum (`Internal` default, `ExternalSync`) carried on every stored entry. Anything ingested from a third-party sync source (Gmail/Slack/Notion/Composio/etc.) **must** be stored via `store_with_taint(..., MemoryTaint::ExternalSync)`, and an unknown/corrupt taint value decodes fail-closed to `ExternalSync` (never silently trusted). This taint gates whether the "subconscious" background loop is allowed to call `external_effect` tools on a turn that touched tainted context â€” i.e. provenance tracking used directly as a tool-execution gate, not just an audit trail. `src/openhuman/memory/traits.rs`.

### Storage layers
Two coexisting tiers, deliberately kept separate per the migration spec:
1. **Substrate tables** (moving to the `tinycortex` crate): `mem_tree_chunks` (+embeddings), `vectors`, `kv_global`/`kv_namespace`, `mem_tree_entity_index`/`entity_edges`/`entity_hotness`, `mem_tree_trees`, `mem_tree_summaries`, `mem_tree_buffers`, `mem_tree_score`, `mem_tree_jobs` (job queue).
2. **Host-retained "UnifiedMemory" namespace-document tier** (explicitly *not* migrating â€” product-specific): `memory_docs`, `graph_global`/`graph_namespace`, `episodic_log`(+FTS5), `event_log`(+FTS5+embeddings), `conversation_segments`/`segment_embeddings`, `vector_chunks`, `user_profile`. Lives in `src/openhuman/memory_store/unified/{init,fts5,events,segments,profile}.rs`.

### "Memory Tree" (headline feature) â€” hierarchical, scored, cascading summarization
`src/openhuman/memory_tree/` (README at `src/openhuman/memory_tree/README.md`). Three tree kinds â€” `Source`, `Global`, `Topic` â€” share one kind-agnostic engine (`tree/bucket_seal.rs` append+cascade-seal, `tree/flush.rs` time-based partial seal, `tree/registry.rs` kind-parameterized `get_or_create_tree`). Leaves accumulate into buckets; when a bucket seals, `summarise.rs` produces the next level's summary text via the chat model (L_n â†’ L_{n+1}), so the tree compresses upward over time instead of growing unbounded. Retrieval (`memory_tree/retrieval/`) exposes agent-facing tools: `walk`/`smart_walk` (deterministic "E2GraphRAG"-style agentic traversal), `drill_down`, `fetch_leaves`, `search_entities`, `query_{source,global,topic}`. The whole tree is also mirrored to disk as Markdown under `{workspace}/memory_tree/content/{chat,episodic,raw,wiki}/` â€” described in the README as an Obsidian-openable vault, so "memory" is never a vector-only black box; every entry has a readable file.

### Admission scoring / decay-consolidation
`src/openhuman/memory_tree/score/` â€” every ingested chunk runs through `score_chunk`: cheap deterministic signals (`signals/{interaction,metadata_weight,source_weight,token_count,unique_words}.rs`, each normalized `[0,1]`) combine into a weighted total (`score/signals/mod.rs::combine`). A three-band gate avoids LLM cost on the obvious cases: total â‰Ą `0.85` â†’ admit without calling the LLM extractor; total â‰¤ `0.15` â†’ drop without calling it; below `0.3` overall â†’ tombstoned (never reaches the chunk store); the borderline band consults an `EntityExtractor` (regex or LLM). Per-signal values are persisted alongside the total specifically "to make admission decisions debuggable" (`score/mod.rs` docstring). A `PRIORITY_TAG`/`PRIORITY_BOOST` mechanism lets the ingest path (e.g. GitHub commit messages, merged PRs) nudge otherwise-marginal content above the bar. This is the closest thing to "decay/consolidation" in the codebase â€” there's no explicit time-decay of stored entries found in this pass, but the cascading tree-summarization *is* the consolidation mechanism (raw leaves get compacted into higher-level summaries as buckets seal on a hotness/time cadence), and entity "hotness" (`mem_tree_entity_hotness`) gates when a topic tree gets spawned at all.

### Per-agent / per-user scoping
- **Per-user**: everything is namespace-scoped (`namespace_summaries`, `RecallOpts.namespace`); "local-authored" namespaces (`working`, `agent`, `local`, `core`, `global`, `default`, `user`, plus `working.*`/`agent.*`/`tree.*` prefixes) are an explicit allowlist â€” anything else is untrusted by default (see prompt-injection defense below).
- **Per-agent**: `agent_memory/` is a dedicated specialist sub-agent ("the memory agent") that navigates the tree to answer questions, combining vector search + keyword search + entity search + tree browse + direct content reads + source listing. It's registered like any other built-in agent: `agent_memory/agent/agent.toml` (tool allowlist, model hint, iteration cap) + `prompt.rs`/`prompt.md` (dynamic system prompt). `src/openhuman/agent_memory/README.md`.

### Prompt-injection defense at the memoryâ†’prompt boundary (top steal-candidate)
`src/openhuman/agent/harness/memory_context_safety.rs` (~250 lines incl. tests) is a small, self-contained, well-documented mitigation: recall mixes trusted (user/agent-authored) and untrusted (connector-synced Gmail/Slack/Notion/etc.) content with identical visual weight in the prompt. `is_potentially_untrusted()` is a **default-deny classifier** (unknown namespace â†’ untrusted; known connector key prefixes like `gmail:`/`slack:`/`notion:` â†’ untrusted even with no namespace) and `wrap_untrusted_for_agent()` wraps flagged content in `<untrusted-source source="...">...</untrusted-source>`, HTML-escaping `&`/`<`/`>` in the body and sanitizing the hint attribute so embedded payloads can't forge or close the marker early. The module docstring is explicit that this is a stopgap ("a proper fix is a typed Provenance enum... out of scope for this commit") rather than a finished system â€” but as a narrowly-scoped, testable, copyable primitive for "external content reaching agent context must be visually and structurally distinguished from instructions," it's directly relevant to Sonik's own host-context/trust-boundary work.

## Integration framework â€” compare to our capability registry

### Controller registry (general capability pattern, not integration-specific)
`src/core/all.rs` â€” a flat `RegisteredController { schema: ControllerSchema, handler: ControllerHandler }` list built once into a `static REGISTRY: OnceLock<Vec<...>>`, validated for duplicates/missing handlers at first access (panics loudly rather than silently shipping a broken registry). Notable: there's a **second, parallel `INTERNAL_REGISTRY`** for controllers that are callable over RPC (e.g. by the Tauri scanner / trusted host callers) but deliberately excluded from agent-facing tool listings and schema discovery â€” an explicit agent-visible vs. host-only capability split baked into the registry itself, not bolted on as a filter later.

### Three-tier integration model
1. **Deep native providers** for ~30 high-value toolkits (gmail, notion, slack, clickup, github, linear, â€¦): `src/openhuman/memory_sync/composio/providers/traits.rs` defines `ComposioProvider` (async trait: `toolkit_slug`, `sync_interval_secs` default `Some(15min)`, `curated_tools` default `None`, `fetch_user_profile`, `sync`, `fetch_tasks` default `Err(...)`, plus trigger/OAuth-handoff hooks). Every opt-out capability (no periodic sync, no task surface, no curated whitelist) is a **sensible trait default**, not a required override â€” adding a shallow provider is cheap, and the trait doc explicitly justifies "a trait, not a giant match" for testability/isolation. Providers register into a process-global `HashMap<toolkit_slug, Arc<dyn ComposioProvider>>` (`providers/registry.rs`), looked up by both the webhook-trigger event-bus subscriber and the periodic sync scheduler.
2. **Generic Composio proxy** for the long tail (README claims "100+ OAuth integrations, 5,000+ MCP servers"): the Rust core **never calls the Composio API directly** â€” `composio/mod.rs` docstring is explicit that a separate TypeScript backend (`src/routes/agentIntegrations/composio.ts`, not in this repo) owns the Composio API key, billing/margin, toolkit allowlist, and HMAC webhook verification; the core only does RPC wrapping, agent-tool exposure, and event-bus fan-out (`composio:trigger` Socket.IO â†’ `DomainEvent::ComposioTriggerReceived`). This is a clean **trust/liability boundary**: third-party API keys and billing never touch the client/core process.
3. **Sandboxed user-authored skills** (the "90,000+ Skills" claim): QuickJS-sandboxed bundles, described only in a code comment as living in the sibling `tinyhumansai/openhuman-skills` repo â€” not present locally, not verified further in this pass.

### Typed connector registry feeding memory
`src/openhuman/memory_sources/` (`README.md` present, thorough) â€” separate from the Composio provider layer above, this is the "what feeds my memory" registry: `MemorySourceEntry` records (Composio connections, local folders, GitHub repos, RSS feeds, Twitter queries, web pages) persisted flatly in `config.toml` under `[[memory_sources]]`, CRUD'd through 9 RPC controllers, with one `SourceReader` trait implementation per `SourceKind` (`readers/{composio,folder,github,rss,twitter,web_page}.rs`) dispatched via `reader_for(kind)`. Composio sources auto-reconcile into this registry on every `list` call and on OAuth-connect, so a freshly-connected integration shows up without a restart. Sync status is computed by querying chunk-count/freshness against the memory store directly (`mem_tree_chunks` prefix match), not tracked as separate state â€” a nice "don't duplicate state you can derive" instinct, called out explicitly in the module's own "Notes/gotchas" section.
