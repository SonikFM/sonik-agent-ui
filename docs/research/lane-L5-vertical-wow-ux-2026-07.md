# Lane L5: Vertical Marketplace Fit + Install→Wow UX

**Scope:** How vertical-operations SaaS platforms run app marketplaces, whether agent-driven setup is claimed territory in those verticals, and how to present a trust/permission gate at install without killing the "wow" of Sonik's guided-canvas-to-real-booking flow.

**Horizon:** Sources are 2025–2026 unless flagged otherwise. Current date context: July 2026.

**Builds on:** Foundation lanes L1–L3 findings — Zapier "Guided Templates" as the best-in-class frictionless-install pattern, and the identified gap (D1/D2) that Sonik shows no trust-tier/permission review at install.

---

## 1. Vertical marketplace profiles + what's structurally different

### Profiles

**Toast (restaurants).** Access runs through a formal Partner Integrations program: prospective partners sign an API Documentation License Agreement, apply via an integration-partner application form, and are reviewed before receiving scoped API access "to read and write data on behalf of shared customers." Toast's own directory lists integration/channel partners; third-party aggregators count roughly 143 apps in the Toast ecosystem. (2 sources: [Toast Partner Application](https://pos.toasttab.com/partners/integration-partner-application), [Toast Integration Partner Directory](https://pos.toasttab.com/partners/directory); app count from [appmarketplace.com Toast listing](https://appmarketplace.com/marketplaces/toast-integrations/), single-source for the specific count — treat "143" as approximate.)

**Square App Marketplace.** Listing requires becoming an approved "Square app partner" first — which itself requires the applicant to already have signed up **at least five active sellers** using the integration before applying. Approval takes up to 14 business days, requires accepting a Partner Integrated Marketplace Agreement (PIMA), and includes mandatory QA testing for technical and usability requirements. ([Square App Marketplace FAQ](https://developer.squareup.com/docs/app-marketplace/faq), [Square App Marketplace Requirements](https://developer.squareup.com/docs/app-marketplace/requirements)) — the "5 active sellers before listing" requirement is a distinctive proof-of-traction gate not typical of horizontal marketplaces.

**ServiceTitan Marketplace (home services).** Runs a three-tier partner program — Silver, Gold, Titanium — where higher tiers unlock deeper co-marketing and account-planning support, not just technical access. As of 2026, ServiceTitan is pushing toward a **full certification milestone: every app in the marketplace must be certified**, following a multi-step evaluation of "functionality, security, and data transparency," with **annual re-certification** required to keep the badge. More than 70% of customer tenants have installed a marketplace app, and 120+ companies applied to join in a recent three-month window. ([ServiceTitan App Marketplace Program Guide](https://www.servicetitan.com/legal/app-marketplace-program-guide), [ServiceTitan: Raising the Bar with App Certification](https://www.servicetitan.com/blog/app-marketplace-relaunch), [ServiceTitan overview for partners](https://help.servicetitan.com/docs/servicetitan-overview-for-app-marketplace-partners) — 3 corroborating sources.)

**Shopify App Store.** The most horizontal-leaning of the group but still enforces a strict technical bar: as of April 2025 all new public apps must use the GraphQL Admin API exclusively, and apps must implement mandatory GDPR webhooks. The elevated "Built for Shopify" badge requires 50+ net installs from paying shops, 5+ reviews, a minimum rating, and a performance budget (no more than a 10-point Lighthouse regression; p95 latency under 500ms at 1,000+ requests/28 days). Review cycles run 2–3 rounds over 4–6 weeks. ([Shopify App Store Requirements](https://shopify.dev/docs/apps/launch/shopify-app-store/app-store-requirements), [Built for Shopify Requirements](https://shopify.dev/docs/apps/launch/built-for-shopify/requirements) — 2 primary sources.)

**Mindbody (fitness/wellness).** Explicitly **hand-curates** rather than auto-lists: "new apps are added intentionally, not automatically," with quality and ease-of-installation as gating criteria. This isn't cosmetic — Mindbody reports customers who use integrations see **10% more revenue, 40% more bookings, and ~35% more client purchases** than non-integration customers, which is the commercial argument the vertical marketplace makes for curation depth. The consumer-facing side (3M+ users in the Mindbody app) and the operator-facing marketplace (700+ partner solutions via open API) are kept distinct. ([Partner Fleet: How Mindbody built the #1 fitness-tech marketplace](https://www.partnerfleet.io/blog/how-mindbody-built-the-number-one-integrations-marketplace-in-fitness-tech) — single primary source for the specific revenue/booking lift stats; directionally corroborated by [Mindbody Integrations](https://integrations.mindbodyonline.com/) but the numbers themselves are single-sourced marketing data and should be flagged as such.)

**Hospitality PMS marketplaces — Cloudbeds and Mews.** Cloudbeds connects 450+ integration partners across POS, CRS, CRM, payments, and guest-service categories, plus an open API for custom connections. Mews takes a different structural approach — a core PMS built partly through acquisition, wrapped in a marketplace ("hospitality's biggest app store") that now exceeds **1,000 marketplace integrations** with no connection fees, plus multiple open APIs (Connector API, Booking Engine API) and, new in 2026, an open API for Mews POS. ([Cloudbeds Marketplace](https://www.cloudbeds.com/integrations/), [Mews surpasses 1,000 Marketplace integrations](https://www.mews.com/en/press/mews-surpasses-1000-marketplace-integrations-with-hospitalitys-biggest-app-store), [Mews Marketplace docs](https://docs.mews.com/getting-started/mews-marketplace) — 3 sources.) Neither vendor publishes a detailed public security-vetting process comparable to ServiceTitan's certification tiers or Salesforce's AppExchange review; comparison sources ([Hotel Tech Report](https://hoteltechreport.com/compare/cloudbeds-myfrontdesk-vs-mews), [Codelevate](https://www.codelevate.com/blog/mews-vs-cloudbeds-which-pms-will-power-your-rental-business-in-2026)) describe architecture differences but not vetting rigor — **flagged as a gap in available evidence**, not a confirmed absence of vetting.

**Reference point outside the vertical set — Salesforce AppExchange.** Useful as a ceiling case: a mandatory Security Review (manual + automated code inspection, architecture document, compliance disclosures) takes 4–6 weeks and costs a flat $999 fee for paid apps. This is a horizontal *enterprise* marketplace, but it shows how far vetting rigor can go when the buyer is risk-averse. ([Salesforce AppExchange Security Review — How It Works](https://developer.salesforce.com/docs/atlas.en-us.packagingGuide.meta/packagingGuide/security_review_how_it_works.htm), [Salesforce Security Review guide 2026](https://magicfuse.co/blog/how-to-pass-salesforce-appexchange-security-review) — 2 sources.)

### What's structurally different about vertical operations marketplaces

1. **Curation is manual and relationship-mediated, not just automated-gate-based.** Mindbody explicitly rejects auto-listing; ServiceTitan's tiering ties technical certification to commercial relationship depth (Titanium = joint account planning, not just "passed QA"). This is a deeper investment than Shopify's largely self-serve, metrics-triggered "Built for Shopify" badge.
2. **Publishers are category incumbents, not generalist developers.** Toast, Square, ServiceTitan, and Mindbody partners are overwhelmingly point-solution vendors *already* selling into that same vertical (payroll, marketing, POS-adjacent hardware) — compare to Shopify or the GPT Store, where a generic developer with no retail/commerce background can plausibly publish a working app.
3. **Install risk is operational, not just data-scope risk.** A bad app in a horizontal dev-tool marketplace misbehaves inside a sandboxed workflow. A bad integration in a vertical operations marketplace can double-book a table, mis-tender a card at a live terminal, or corrupt a tee-sheet mid-shift — the "blast radius" of a broken install is a real-time operational failure, not a data leak. This is the throughline for ServiceTitan's move to 100% certification and Square's "5 live sellers before you can list" proof-of-traction bar — both are structurally responding to install-into-live-operations risk, not just code quality.
4. **Operator trust is still partly pre-digital / relationship-based.** Independent consultancies (e.g., Bearheart) vet POS integration vendors on restaurant operators' behalf and coordinate installs manually — a reminder that in this vertical, "trust" isn't fully solved by a marketplace badge; a nontrivial share of operators still outsource vetting to a human advisor. ([Bearheart Toast POS Consultant](https://bearheartconsulting.com/hospitality-technology-services/point-of-sale-implementation/toast-pos-consultant/) — single source, but consistent with the broader finding that these are high-touch B2B relationships, not self-serve app stores in the consumer sense.)

## 2. White-space assessment: is agent-driven vertical setup unclaimed?

**Short answer: yes, as of July 2026, with reasonable confidence but not certainty.** Multiple independent search angles turned up no vendor combining (a) a vertical operations marketplace with (b) agent-led, self-service setup that (c) ends in a real, live operational action.

Evidence for this:

- **None of the six vertical platforms profiled above (Toast, Square, ServiceTitan, Shopify, Mindbody, Cloudbeds, Mews) publish an agent-driven install/configuration experience.** Searches targeting exactly this combination returned only generic AI-feature coverage (Mindbody's "AI front desk," ServiceTitan's 2026 marketplace *browsing* refresh) — not agent-assisted app setup. ([ServiceTitan marketplace refresh](https://www.servicetitan.com/blog/app-marketplace-relaunch), [Mindbody AI front desk](https://www.mindbodyonline.com/))
- **Horizontal AI-agent marketplaces exist but are not vertical-specific and don't combine setup with install.** The current landscape (Anthropic Claude Skills/MCP Connectors Directory, OpenAI GPT Store, Hugging Face, Replit, LangChain, Vercel, Cloudflare) is explicitly general-purpose distribution — one industry survey of this landscape found zero mentions of vertical-industry marketplaces or combined setup+install experiences. ([AI Agent Marketplaces 2026: Discovery and Distribution](https://www.digitalapplied.com/blog/ai-agent-marketplaces-2026-discovery-distribution) — single source for this specific negative finding, but consistent with the OpenAI and Anthropic primary docs reviewed separately: [OpenAI App Submission Guidelines](https://developers.openai.com/apps-sdk/app-submission-guidelines), [Claude Connectors Directory](https://support.claude.com/en/articles/11175166-get-started-with-custom-connectors-using-remote-mcp).)
- **The closest direct competitor undercuts, rather than confirms, a claimed space.** RebootAI ships native connectors for exactly this stack — Toast, Square, Mindbody, ServiceTitan, Shopify, Stripe — and automates real operational tasks (invoicing, scheduling, bookings). But its setup model is **human-led, not self-service**: a 30-minute founder audit call, a 2–4 week engineering build-out, and a 2–4 week "shadow mode" period where the agent's would-be actions are reviewed before it's allowed to act live. ([RebootAI](https://rebootai.com/) — single source, marketing site; treat directional claim as credible but unverified by a third party.) This is materially different from Sonik's model: RebootAI substitutes a sales/services motion for self-serve configuration, and its trust mechanism is a multi-week shadow period rather than a real-time preview-and-approve gate.
- **Consumer-facing agentic booking is advancing fast but sits on the other side of the relationship.** Google's AI Mode Canvas with Project Mariner and Booking.com's agentic AI tools let *travelers* book via natural language across OpenTable, Resy, Ticketmaster, and similar partners — but these are demand-side booking agents, not operator-side setup/configuration tools for the business running the venue. ([Google AI Mode Canvas](https://blog.google/products-and-platforms/products/search/agentic-plans-booking-travel-canvas-ai-mode/), [Booking.com agentic AI](https://news.booking.com/bookingcom-debuts-agentic-ai-innovations-adding-to-its-robust-suite-of-genai-tools-for-customers/) — 2 sources, consistent framing.)

**Net:** agent-driven, self-service *setup* inside a vertical operations marketplace — where "ask in plain language" leads directly to a real booking/reservation with a receipt, without a human sales call or a multi-week shadow period — appears genuinely unclaimed. Confidence is moderate-high rather than absolute: this rests on the absence of evidence across ~10 targeted searches plus one direct competitor teardown, not an exhaustive audit of every vertical SaaS vendor.

## 3. Install→wow UX patterns (NNG heuristic lens)

Applying Nielsen Norman Group's method — progressive disclosure, recognition over recall, user control and freedom, error prevention:

- **Progressive disclosure has a hard ceiling.** NNG's core guidance: show only the most-needed options first, defer the rest — but **designs with more than two disclosure levels typically fail**, because users lose track of what layer they're on. If a flow needs three-plus tiers, the fix is to simplify the design, not to add a third layer of "advanced" settings. ([NN/g: Progressive Disclosure](https://www.nngroup.com/articles/progressive-disclosure/)) This directly bounds how many layers Sonik's guided canvas can defer before it stops helping and starts confusing.
- **Progressive disclosure measurably improves three of the five usability components** — learnability, efficiency of use, and error rate — by supporting recognition over recall rather than demanding users hold context in their heads. (Same source.)
- **Contextual, behavior-triggered disclosure outperforms linear onboarding substantially.** One large-scale dataset (15M interactions, Chameleon) found linear onboarding completes at 53%, versus 75% for contextual/behavior-triggered disclosure, with a 30% lift in paid conversion. **Single-source figure** — directionally consistent with the general onboarding literature reviewed, but the specific percentages come from one vendor's dataset and should be treated as illustrative, not definitive. ([via UXPin's progressive disclosure roundup](https://www.uxpin.com/studio/blog/what-is-progressive-disclosure/))
- **Time-to-value has a real ceiling for "wow."** Best-in-class SaaS delivers first value in 2–5 minutes; the "aha moment" should land under 5 minutes even if full mastery unfolds over days. Checklists with visible progress bars roughly double completion (35% → 67%) versus checklists without progress indicators. ([Best onboarding UX 2026 roundups](https://userpilot.com/blog/best-user-onboarding-experience/), [SaaS onboarding best practices 2026](https://peppereffect.com/blog/saas-onboarding-best-practices) — 2 broadly consistent secondary sources, though exact percentages vary by source and should be read as directional, not precise.)
- **Error prevention and user control/freedom converge in the "Intent Preview" pattern**, which is the sharpest analogue to Sonik's existing preview-before-approval step (detailed in Section 4 below, since it's also the load-bearing pattern for the permission question).

## 4. Presenting trust/permissions at install without killing the wow

This is where the industry has moved fastest in the last 12 months, and where the reconciliation for Sonik's gap actually lives.

**a) All-or-nothing consent is being retired industry-wide, not just discussed.** Google is mid-rollout of a **granular OAuth consent screen** — replacing the old single "Allow" button with per-permission checkboxes — with Phase 3 for published web apps and Workspace add-ons starting January 7, 2026, and granular consent for Chat apps built on Apps Script starting January 20, 2026. This confirms scannable, itemized permission review is now a baseline user expectation, not a novel ask. ([Google Workspace Updates: Granular OAuth consent](https://workspaceupdates.googleblog.com/2026/01/granular-oauth-consent-google-chat-apps.html), [Configure OAuth consent screen](https://developers.google.com/workspace/guides/configure-oauth-consent) — 2 primary Google sources.)

**b) Just-in-time / incremental consent is the named pattern for reconciling permission asks with flow.** Rather than front-loading every permission the app might ever need, just-in-time consent asks for a specific permission "at the point in time when someone needs to complete a task" — i.e., the permission ask is triggered by the action, not by installation. ([IF Design Patterns: Just-in-time consent](https://catalogue.projectsbyif.com/patterns/just-in-time-consent/)) This is corroborated by the OpenAI app model, where users are shown what data may be shared with a third-party app *when they connect it*, with a visible disconnect option at any time. ([OpenAI App Submission Guidelines](https://developers.openai.com/apps-sdk/app-submission-guidelines))

**c) Progressive autonomy tiers are emerging as the standard trust-expansion model for agents specifically.** A three-mode pattern — **Suggest** (agent proposes, human approves every action) → **Co-pilot** (agent acts on routine tasks, asks for anything consequential) → **Autopilot** (agent acts, reports after) — lets the system earn expanded autonomy through a track record of correct approvals, rather than demanding full trust on day one. NNG itself is reportedly naming "trust, not capability" as the central AI design problem of 2026. ([Designing for AI Agents: 10 UX Patterns (2026)](https://mantlr.com/blog/designing-for-ai-agents-ux-patterns-2026), [Agentic UX pattern framework](https://aiuxplayground.com/frameworks/agentic/) — 2 sources; the specific NNG attribution is embedded in the Mantlr piece and should be treated as secondhand pending direct NNG confirmation.)

**d) The "Intent Preview" pattern is the mechanism that makes a permission ask and an action preview the same screen instead of two sequential gates.** Its core discipline, directly relevant to Sonik's existing preview step:
- The preview must be **rendered from the same plan data the executor will run** — never a "cosmetic" summary that could diverge from what actually executes. A preview that doesn't match execution is "an alibi, not informed consent."
- Each step is tagged **reversible or irreversible**, and the preview **never auto-dismisses**.
- Users can **edit individual steps** before approving — approval is a structured review, not a binary yes/no.
- Use it exactly when actions are consequential/hard to reverse and the agent inferred intent rather than following an explicit instruction — which is precisely Sonik's situation (a plain-language ask, translated into a concrete booking command).
([AI UX Design Guide: Intent Preview](https://www.aiuxdesign.guide/patterns/intent-preview) — single source, but its five implementation rules are internally well-reasoned and consistent with the separately-sourced granular-consent and human-in-the-loop material above.)

**e) Receipts are becoming the standard trust-closing artifact across agentic transactions generally**, not a Sonik-specific idea — which validates rather than undercuts Sonik's current mechanic. In agentic commerce, "Payment Receipts close the audit loop with a merchant confirmation ID, providing non-repudiable proof of execution," and Juniper Research (April 2026) names **trust as the #1 barrier to agentic commerce adoption — ahead of every technical concern**. ([Retail TouchPoints: From Checkout to Trust Layer](https://www.retailtouchpoints.com/executive-viewpoints/from-checkout-to-trust-layer-how-merchants-can-prepare-for-agentic-commerce/618997/), [Juniper Research: Agentic Commerce $1.5T by 2030](https://www.juniperresearch.com/press/agentic-commerce-set-to-generate-15-trillion-globally-by-2030-as-payments-infrastructure-leaders-revealed/) — 2 independent sources corroborate both the receipt pattern and the trust-as-primary-barrier finding.)

**f) Trust badges as a lightweight, disclosed signal (rather than the full audit) are precedented at exactly the granularity Sonik needs.** Anthropic's Claude plugin/connector directory uses a two-tier model — plugins that clear only automated checks vs. an **"Anthropic Verified"** badge for plugins that pass deeper review — giving users a scannable trust cue without requiring them to read an audit report. ServiceTitan's "Certified" badge and Salesforce's Security Review badge work the same way in the vertical-SaaS world: the badge *is* the interface for trust, not a substitute for it. ([Claude Code Plugins Directory](https://www.techtimes.com/articles/317139/20260525/claude-code-plugins-get-official-directory-anthropic-flags-unverified-mcp-risks.htm), [ServiceTitan certification](https://www.servicetitan.com/blog/app-marketplace-relaunch) — 2 sources, consistent pattern across both a horizontal AI platform and a vertical operations platform.)

**The reconciliation principle:** the friction of a permission gate and the delight of a first-run preview don't have to be sequenced (permission screen, *then* preview screen) — they can be **merged into one screen**: the same card that shows "here's the exact reservation I'm about to create" also shows, inline, the specific scope this action requires and a lightweight trust indicator, sourced from the identical execution plan. First-time use of a given action category shows slightly more explanation; repeat use of the same action collapses to a lighter confirm, because trust has already been demonstrated (progressive autonomy) — echoed by Zapier's own locked/editable field split, which is really a form of progressive disclosure baked into a setup wizard rather than a separate permission gate. ([Zapier Guided Templates](https://help.zapier.com/hc/en-us/articles/43465487495181-Guided-templates))

## 5. Recommended install→value flow for Sonik

Given Sonik's current mechanic (plain-language ask → guided canvas → preview of a concrete command → approval → real booking + receipt), no existing marketplace browse/install surface, and the requirement (foundation D1) to add a human trust gate without breaking flow, the shortest path from "user wants X" to "real outcome created" is:

1. **Ask, no catalog browsing required.** The user states intent in plain language. Sonik should *not* force a marketplace-style "browse and install an app" step before this — defer the marketplace concept until after the agent has already demonstrated value once, mirroring Zapier's "start from a working template" over "browse a catalog first."
2. **Guided canvas, progressive disclosure capped at two levels.** The agent pre-fills everything it can infer; only genuinely required fields surface, each with inline contextual help (Zapier's locked/editable-field pattern). Do not add a third disclosure tier ("advanced settings") — NNG's evidence says that's where usability breaks down.
3. **One combined Preview + Trust screen — this is the load-bearing change.** Show the concrete command (what will actually be booked/reserved) and the specific action-tier/permission being invoked for *this* action side by side, rendered from the same execution plan that will run (no cosmetic preview), with the action marked for reversibility. On a user's *first* use of a given action category (e.g., "create a live tee-sheet reservation"), show a slightly fuller trust explanation of what system this touches; on repeat use of that same category, collapse to a lighter-weight confirm — this is the progressive-autonomy pattern applied per-action-type rather than as a one-time global gate.
4. **Approve → real action executes → visible receipt with a confirmation ID.** This already matches where the broader agentic-commerce industry is converging (receipts as the standard trust-closing artifact) — keep and strengthen this, since it's simultaneously Sonik's existing differentiator and the emerging industry norm.
5. **A persistent, lightweight "what my agent can do" surface (not a full marketplace) as the next investment.** Rather than building install/browse UI first, ship a simple dashboard showing granted action tiers, when each was last used/reviewed, and a one-click way to step an action category back down to "ask every time." This closes the permission-accumulation risk NNG and the agent-UX literature both flag, and it doubles as the seed of a future marketplace surface without requiring Sonik to build app-store browsing before it has proven the agent-driven setup wow.

This flow keeps the trust gate from ever feeling like a second, separate hoop — it's folded into the same screen that already delivers the "wow" (seeing the exact real-world action about to happen), and it only asks for more explanation the first time a given category of real-world consequence is at stake.

## So-what bullets

- **The unclaimed territory is narrow but real:** none of the six vertical operations marketplaces studied (Toast, Square, ServiceTitan, Shopify, Mindbody, Cloudbeds/Mews) offer agent-driven setup, and the nearest direct competitor (RebootAI, same exact tool stack) still relies on a human sales call and a multi-week shadow-mode trust ramp rather than self-serve preview-and-approve — Sonik's model is differentiated on exactly this axis today.
- **Vertical marketplaces are structurally higher-trust environments than dev-tool marketplaces** because a broken install here breaks live revenue operations (a double-booked table, not a sandboxed API call) — this justifies investing in a real trust gate rather than copying a lightweight horizontal app-store pattern wholesale.
- **The industry has already solved "permission gate without killing flow" — the answer is fusion, not sequencing:** granular just-in-time consent (Google's 2026 OAuth redesign), Intent Preview's "same-plan" discipline, and progressive-autonomy tiers all point to merging the permission ask into the action preview itself, not stacking a separate consent screen in front of it.
- **Sonik's receipt mechanic is already aligned with where agentic commerce is converging industry-wide** (receipts as the non-repudiable trust-closing artifact) — this is a keep-and-sharpen finding, not a gap to fix.
- **Progressive disclosure has a hard design ceiling (two levels, per NN/g)** — the guided canvas should resist adding a third "advanced" tier even under pressure to support power users; NNG's evidence says a flow that needs three tiers should be redesigned, not deepened.
