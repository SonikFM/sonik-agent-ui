## What it is

Odysseus is a self-hosted, single-tenant "AI workspace" app: chat + tool-using agents, MCP client, deep research, a document editor, email (IMAP/SMTP), notes/tasks/calendar (CalDAV), an image gallery, and a hardware-aware local-model server ("Cookbook"). It's a FastAPI backend (`app.py`, 1,145 lines) with a large flat Python source tree (`src/` — 95 files, `routes/` — 54 files) and a vanilla JS/CSS frontend (`static/`, no build step — `static/style.css` alone is 36.6k lines). It ships as Docker Compose (app + ChromaDB + SearXNG) or a native install/launcher script per OS. This is a solo/small-team hobby-scale project ("I don't know what I'm doing, help" — ROADMAP.md), not an enterprise SDK — the value here is in specific runtime patterns, not the project as a whole.

## Relevance verdict

**Worth studying, not worth vendoring.** Odysseus is AGPL-3.0-or-later. Copying any of its source into Sonik (a closed commercial product) would create copyleft exposure if that code ships in a networked service — treat every file below as **read-for-technique-only**: reimplement the pattern in Sonik's own words/architecture, don't paste code. With that caveat, several runtime modules are directly relevant to the agent-marketplace architecture work (`[[agent-marketplace-architect-lane]]`, D001–D017): Odysseus has already built, in production code, several primitives that lane is still contract-only for — most notably self-improving skill capture and a denylist-from-allowlist tool-gating pattern for privilege modes. It's a useful "what does this look like once someone actually ships it" reference.

## Steal-candidates (read for technique, reimplement don't copy)

**Skill capture / marketplace-adjacent primitives**
- `services/memory/skill_extractor.py` (305 lines) — background LLM pass that watches agent runs (>=2 rounds or >=2 tool calls) and asks a model to distill a *reusable, on-computer* procedure into a skill, conservatively (`return null` unless it's a genuine repeatable method — explicitly excludes one-off/personal/outside-the-computer work). This is close to what a "learned skill → marketplace package" pipeline in Sonik would need for auto-authoring candidate packages from real usage.
- `services/memory/skills.py` (716 lines) + `services/memory/skill_format.py` (444 lines) — skills stored as `data/skills/<category>/<name>/SKILL.md`, YAML frontmatter + structured markdown body (When to Use / Procedure / Pitfalls / Verification), owner-scoped via `owner:` frontmatter, usage counters in a separate sidecar JSON so the SKILL.md content doesn't churn on every retrieval read. That "content vs. usage-metadata are separate files" split is a clean pattern worth reusing conceptually for any file-backed capability registry.
- `services/memory/skill_importer.py` (283 lines) — importer for skill packages from elsewhere; worth a closer read if Sonik ever needs an "import a skill/package from disk" flow.
- `services/memory/memory_extractor.py` (659 lines) — background fact extraction after each LLM turn, PLUS a periodic LLM-driven audit that consolidates duplicates, rewrites vague entries, and deletes junk, short-circuited by an order-independent SHA-256 fingerprint of the memory set (`_fingerprint_entries`) so the audit skips re-running the LLM when nothing changed since the last clean pass. That fingerprint-gated re-audit trick is generically useful anywhere an expensive LLM pass should be idempotent-skipped.
- `src/memory_provider.py` (320 lines) — a clean `MemoryProvider` ABC (`remember`/`recall`/`list_memories`/`delete` + optional `get_tool_schemas`/`handle_tool_call`) with a `MemoryProviderRegistry` that composes native + external memory providers and raises on tool-name collisions between providers. This is a good shape reference for a pluggable-provider registry pattern generally (not memory-specific).

**Tool-gating / privilege-mode enforcement**
- `src/tool_security.py` (266 lines) — Plan Mode is implemented as an *allowlist* (`PLAN_MODE_READONLY_TOOLS`, read-only tools only) but enforced through the agent loop's existing *denylist* gate, by inverting: `all_known_tool_names - allowlist = denylist`. Known tool names come from the function-schema registry, backstopped by a static `_PLAN_MODE_KNOWN_MUTATORS` set so an unregistered/XML-only tool or a failed schema import still fails closed rather than silently leaving a mutator enabled. This "derive a safe denylist from a smaller allowlist, with a static backstop for fail-closed behavior" pattern is directly applicable to any Sonik surface that needs a read-only/plan/preview mode layered on top of an existing tool-execution gate (relevant to the trusted-approval doctrine work).
- `src/tool_policy.py` (242 lines) — per-turn `ToolPolicy` dataclass composing disabled/hidden tools + a regex-detected "guide-only mode" (user says "don't use tools") that forces `block_all_tool_calls=True` and disables MCP for that turn. Small, clean, worth the read for the regex-based intent-detection-to-policy pattern.
- `src/task_action_policy.py` (47 lines) — a tiny, single-purpose "is this scheduled-task action admin-only" check (`run_local`, `run_script`, `ssh_command`, `cookbook_serve`) with owner-privilege resolution that checks an internal-tool-user sentinel, then falls back to `AuthManager.is_admin`. Worth noting as a minimal reference for gating dangerous scheduled-task action types by owner privilege.

**Agent-loop orchestration**
- `src/agent_loop.py` (4,529 lines — huge, read narrowly, not wholesale):
  - `_run_verifier_subagent` (~line 2408) — a *second*, fresh-context model call that reads only the user's original request + a snapshot of actions taken (no shared history) and judges completion against a checklist, returning `VERIFICATION: FAIL: <reasons>` or nothing. The stated rationale — "a model checking its own work rationalizes; one that didn't do the work reads it cold" — is a good verification-agent pattern for Sonik's own `verifier`/`code-reviewer` agent design.
  - `_detect_runaway_call` (~line 2529) — counts identical `{tool_type}:{content[:120]}` call signatures and flags a runaway loop only past a repeat threshold (15), so a legitimate batch of 18 distinct calendar-event creations isn't false-flagged. Simple, effective loop-guard technique.
  - `build_active_plan_note` (~line 2504) — re-injects the full approved plan as a system note every turn so a long plan on a weak/small-context model survives history truncation, paired with an `update_plan` tool call convention for progress checkboxes.
  - `TOOL_SECTIONS` (~line 349) + `_assemble_prompt` (~line 617) — modular per-tool prompt sections assembled conditionally based on which tools are active for the turn, with a `compact` mode — a reasonable answer to the "agent prompt/context bloat" problem the ROADMAP explicitly calls out as unsolved for small local models.
- `src/context_compactor.py` (456 lines), `maybe_compact` (~line 312) — token-budget-triggered compaction: splits system vs. conversation messages, summarizes the older half via a (separately configurable) utility model, keeps the recent half verbatim, tracks compaction count via `[Conversation summary` markers in prior system messages. Standard technique, cleanly isolated in one function — good as a compaction-flow reference.
- `src/context_budget.py` (82 lines) — adaptive input-token budget: honors an explicit user-set budget (clamped to the model's context window), otherwise auto-scales to 85% of the discovered context window capped at 200k, falling back to a conservative 6000-token default when the window is unknown. Small, well-isolated, pure/testable — worth reading for the "auto vs. explicit" sentinel-value design (`budget_is_explicit` keys off value, not settings-presence, specifically to survive a settings-save path that materializes defaults into JSON).
- `src/agent_runs.py` (213 lines) — detaches an agent/chat SSE stream from the HTTP connection: a background asyncio task drains the generator into a per-session replay buffer; SSE clients subscribe/replay-then-live; closing the tab doesn't kill the run; reconnect within a 180s grace window replays from the buffer. Directly relevant if Sonik ever needs agent runs to survive client disconnect/tab-close (a common gap in chat-agent UIs).
- `src/event_bus.py` (119 lines) — minimal event-triggered task firing: counts events per scheduled task against a `trigger_count` threshold, persists the trigger (`next_run = now`) before handing off to the async scheduler so a mid-flight process restart doesn't lose the trigger after the counter has already reset. Simple event→task wiring worth a glance.
- `src/interactive_gate.py` (203 lines) — a foreground-activity gate: background jobs (schedulers, email pollers) wait for the UI to go quiet (default 1.5s) before running, tracked via active-request counts + a recent-browser-heartbeat window + an active-chat-stream check, with an `asyncio.Condition` for efficient waiting instead of polling. Good reference if Sonik wants background agent work to defer to live user interaction on shared local compute.
- `src/task_scheduler.py` (2,627 lines, `class TaskScheduler` at line 333) — the general scheduled/recurring-task engine (cron + interval + event triggers, LLM vs. action task types). Large; only worth a deep read if Sonik is building its own task-scheduling surface.

**MCP**
- `src/mcp_manager.py` (674 lines, `mcp_tool_is_readonly` at line 105) — classifies MCP-server-declared tools as read-only by heuristic, feeding the same plan-mode allowlist logic above.
- `src/builtin_mcp.py` (340 lines) — auto-registers a small set of built-in MCP servers (spawned via npx), with npm-cache-presence checks (`_is_npx_package_cached`) so a cold-cache first-run doesn't silently hang waiting on npx to fetch a package. `mcp_servers/memory_server.py` and `mcp_servers/rag_server.py` are worked examples of exposing memory and RAG document management as MCP tool servers.

**Not relevant / skip**
- The bulk of `routes/` (54 files) and `src/tool_implementations.py` (4,032 lines, the single largest file in the repo per its own architecture-inventory doc, `specs/architecture-runtime-inventory.md`) are product-surface glue (email, calendar, gallery, Cookbook model serving) specific to Odysseus's own feature set — not agent-marketplace-shaped.
- Frontend (`static/`) is a large unbundled vanilla JS/CSS app (36.6k-line single CSS file, 9.7k-line single JS file for the document editor) — not a pattern worth importing; Sonik's Svelte/design-system stack is already more disciplined here.
- Cookbook (local model hardware-fit/serving) and the email/calendar/CalDAV subsystems are deep, self-contained product features with no marketplace-architecture angle.

## Bring-up

**Runbook: yes**, on Apple Silicon macOS, via the native launcher (Docker on macOS can't reach the Metal GPU, so the setup guide itself recommends native for Apple Silicon).

- **Fastest path**: `git clone https://github.com/pewdiepie-archdaemon/odysseus.git && cd odysseus && ./start-macos.sh` (or use the already-present local clone at `/Users/danielletterio/Documents/GitHub/odysseus` directly). The script reads `.env` for `APP_BIND`/`APP_PORT` overrides, checks/bootstraps a Python 3.11+ venv, and fails fast with a clear message if the target port is already bound.
- **Port**: defaults to **7860**, not the Docker-path default of 7000 — the script comment explicitly notes macOS AirPlay Receiver holds port 7000 on Apple Silicon. Override with `ODYSSEUS_PORT=7900 ./start-macos.sh` if 7860 is also taken.
- **First login**: an admin account (`admin` unless `ODYSSEUS_ADMIN_USER` is set) is auto-created; the temporary password is printed to the terminal on first run.
- **What Dan must provide**: nothing is strictly required to see the UI — auth/session/settings pages work standalone. To exercise chat/agents meaningfully he'd need either a local model endpoint (Ollama, reachable via `LLM_HOST`/`OLLAMA_BASE_URL`) or an API key for a hosted provider entered in Settings after first login. Email/calendar/search features need their own IMAP/CalDAV/SearXNG config but are not needed to see the core chat/agent UI.
- **RAM/hardware**: the app process itself is lightweight per the setup guide ("local model serving is the heavy part"); with no local model served, expect normal Python/FastAPI footprint (low, sub-1GB). If Dan wants to actually serve a local model via Cookbook, RAM/VRAM needs scale with the model chosen — the guide has no fixed number, it's model-dependent.
- **Docker alternative** (`docker compose up -d --build`, port 7000, binds to `127.0.0.1` by default) also works on Apple Silicon but loses GPU access for Cookbook — fine if the goal is just to see the UI, not to serve local models.
- Existing visual reference already in the repo, no capture needed: `docs/odysseus-browser.jpg` (static screenshot) and a hover-to-play demo tour at `docs/index.html`.
