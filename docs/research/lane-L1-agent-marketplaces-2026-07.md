# Lane L1 — Reference Implementations of Agent/AI-App/Tool Marketplaces (as of July 2026)

Status: draft for team-lead synthesis · Search horizon: prioritized 2025–2026 sources · Posture: THOROUGH (load-bearing claims need ≥2 independent sources or are explicitly flagged single-source)

Scope: what the category-defining agent/app/tool marketplaces look like in mid-2026 — installed unit, discovery/install UX, permission/trust model, monetization, adoption — then stress-tested against Sonik's baseline (one package envelope with kinds `bundle/app/workflow/skill/command_tool_pack/agent/artifact_template/mcp_addon/provider_integration`; installs target immutable `packageVersionId`; JSON-render apps are execution-inert, renderer collects input, a trusted controller commits via host-signed approval; permission grants are `off/ask/allow`).

## 1. Per-player profiles

### 1.1 OpenAI — ChatGPT App Directory / Apps SDK

- **Installed unit:** an "app" = an MCP server (tools/backend logic) + an optional UI rendered in a **sandboxed iframe** inside the chat. No UI is required — many apps are tool-only. [Apps SDK docs](https://developers.openai.com/apps-sdk); [launch post](https://openai.com/index/introducing-apps-in-chatgpt/).
- **Discovery/install:** an in-product **App Directory** launched **December 18, 2025**, organized into Feature/Lifestyle/Productivity categories (Booking.com, Spotify, Dropbox, etc. at launch). First use of an app triggers a **connect prompt** disclosing what data may be shared before any tool call executes. [OpenAI dev-submission post](https://openai.com/index/developers-can-now-submit-apps-to-chatgpt/); [Engadget](https://www.engadget.com/ai/openai-just-launched-an-app-store-inside-chatgpt-133049586.html).
- **Permission/trust model:** submission review against "quality, safety, and policy standards" gates publication; per-tool-call approval plus the initial connect-consent is the user-facing control. Two independent sources corroborate the connect-then-approve flow (OpenAI's own post + Engadget's launch coverage).
- **Monetization:** tiered and deliberately conservative. GA/recommended = **external checkout** (redirect to the developer's own site). A narrower **in-app checkout with already-saved payment methods** is allowed with no redirect. A **ChatGPT Payment Sheet** for collecting new payment methods in-chat is **private beta, select partners only**, built on the **Agentic Commerce Protocol (ACP)**, co-developed with Stripe, requiring PCI DSS Level 1-certified PSPs (Adyen, Checkout.com, Fiserv, PayPal, Stripe, Worldpay). [Monetization docs](https://developers.openai.com/apps-sdk/build/monetization); [Buy it in ChatGPT](https://openai.com/index/buy-it-in-chatgpt/). A cited **~4% take rate** on Instant Checkout sales appears in secondary commerce-analysis coverage only — **single-sourced, flagged low-confidence**, not confirmed on OpenAI's own docs.
- **Notable 2026 wrinkle:** OpenAI has since **scaled back** in-chat commerce ambition, revising Instant Checkout to redirect purchases back through merchant storefronts rather than complete them fully in-chat — a signal that in-agent commerce trust/liability remains unresolved even for the best-resourced player. [Futunn coverage](https://news.futunn.com/en/post/69633148/openai-scales-back-chatgpt-s-e-commerce-ambitions-posing-further) (single primary source found; treat directionally, not as precise figures).
- **Adoption:** ChatGPT itself: ~900M weekly active users per OpenAI's own Q1 2026 update, reported as crossing **1B monthly active users by June 2026**; ~2.5B prompts/day; 93% of Fortune 500 using it in some capacity; 50M+ paid subscribers; ~$25B annualized revenue. [OpenAI Q1 2026 signals](https://openai.com/signals/research/2026q1-update/); [aggregator cross-check](https://technologychecker.io/blog/chatgpt-statistics). Note: WAU/MAU figures come from different snapshots in time and are not strictly comparable — treat as a growth trajectory, not one number.

### 1.2 Anthropic — MCP Registry, Connectors Directory, Claude Code Plugins

Anthropic does not run one unified "app store" — it runs a **protocol** (MCP) plus several separately-curated **directories** on top of it. This is architecturally distinct from every other player in this report.

- **Installed unit:** varies by directory — MCP servers (protocol-level tools), Skills (instruction folders), Claude Code plugins (55+ official / 72+ community, per one community source, **single-sourced**), and "Connectors" (Anthropic's own vetted directory, reported at 343 integrations by a third-party GitHub index — **single-sourced, unofficial**).
- **Governance shift (verified, 2 sources):** on **December 9, 2025**, Anthropic donated MCP to a new **Agentic AI Foundation (AAIF)**, a directed fund under the **Linux Foundation**, co-founded with **Block** and **OpenAI**, with Google, Microsoft, AWS, Cloudflare, and Bloomberg backing it. MCP governance itself is unchanged (maintainers keep steering), but stewardship moved to a neutral third party specifically to keep the protocol "open-source, community-driven, and vendor-neutral." [Anthropic announcement](https://www.anthropic.com/news/donating-the-model-context-protocol-and-establishing-of-the-agentic-ai-foundation). This is the only case in the survey of a marketplace-adjacent standard being deliberately de-coupled from its single-vendor originator.
- **Discovery/trust:** the official registry (registry.modelcontextprotocol.io) is **metadata-only** — it points to npm/PyPI/NuGet/Docker Hub, it does not host or vet code. Anthropic's own Software Directory Policy requires developers to supply privacy policies, contact info, docs, and test accounts, and states Anthropic reviews submissions for "safety, security, and compatibility" — but the policy **does not define user-facing installation/consent mechanics**, focusing instead on developer obligations. [Anthropic directory policy](https://support.claude.com/en/articles/13145358-anthropic-software-directory-policy). The MCP spec explicitly recommends treating **remote MCP servers with the same skepticism as any third-party web service**, while local servers are implicitly more trusted. Registry scale: **10,000+ active public MCP servers** as of the most recent count (one earlier figure of ~9,400 appears in older secondary sources — treat 10,000+ as current). [MCP registry](https://registry.modelcontextprotocol.io/); [registry preview post](https://blog.modelcontextprotocol.io/posts/2025-09-08-mcp-registry-preview/).
- **Consumer-surface permission UX (Claude in Chrome / Claude Cowork):** not a marketplace, but a relevant pattern — users can **pre-approve a batch of actions before a run starts** (review the plan upfront, then let it execute without per-step prompts), while Claude still interrupts for irreversible/high-risk actions like purchases. [Claude in Chrome](https://support.claude.com/en/articles/12012173-get-started-with-claude-in-chrome); [Cowork safety](https://support.claude.com/en/articles/13364135-use-claude-cowork-safely). This "batch pre-approval + hard stops for irreversible actions" pattern is a middle ground between per-call approval and blanket allow.
- **Caveat:** a claim that "MCP 2.4 added MFA for high-risk tool calls, an Admin Console, and audit logs" appeared in only one SEO-style aggregator and could not be corroborated against a primary spec source — **flagged as unverified, excluded from confident findings**.

### 1.3 Salesforce — Agentforce + AgentExchange

- **Installed unit:** three tiers unified into one marketplace as of the 2025–2026 rebrand: **apps** (10,000+, ex-AppExchange), **agents/sub-agents** (1,000+ Agentforce agents), and **tools/MCP servers**, plus 2,600+ Slack apps folded in. [AgentExchange](https://agentexchange.salesforce.com/); [Salesforce Ben](https://www.salesforceben.com/appexchange-slack-marketplace-and-the-agentforce-ecosystem-are-now-one-with-fresh-50m-funding/).
- **Discovery/install:** single unified marketplace surfacing 186,000+ customer reviews to inform purchase decisions (vendor-reported figure, corroborated across two Salesforce-adjacent sources).
- **Permission/trust model:** every listing undergoes a **mandatory security review** ("deep-dive technical audits and vulnerability scans" per Salesforce Ben) before listing. Post-install, admins retain control over **data permissions, API scopes, and integration boundaries at user/role/profile granularity**. [SalesforceDevOps.net analysis](https://salesforcedevops.net/index.php/2026/04/14/agentexchange-salesforces-bet-that-trust-can-scale-with-agentic-speed/).
- **Monetization:** a **$50M AgentExchange Builders Initiative** for ISV partners — capital, an "Approved Forward Deployed Engineering Partner Network," lead referrals, and marketing development funds, explicitly aimed at smaller shops.
- **Explicit gap acknowledged by an independent analyst covering the launch:** *"Who decides when an agent's behavior drifts after deployment? Who is liable when a sub-agent exfiltrates data through a tool call that was approved at install time? These questions are not in the press release."* — a direct critique that install-time review does not cover runtime drift.
- **This gap is not hypothetical — it already happened.** **ForcedLeak** (CVSS 9.4), discovered by Noma Security and disclosed publicly in **September 2025**, was an **indirect prompt-injection** attack against Agentforce: attackers embedded malicious instructions in the 42,000-character Web-to-Lead "Description" field, which Agentforce processed and executed without distinguishing attacker-supplied data from legitimate instructions. A **content-security-policy allowlist entry pointing to an expired domain** let the attacker re-register that domain for ~$5 and use it as a "trusted" exfiltration channel for CRM lead emails via an `<img>` tag. Salesforce patched with Trusted URL Enforcement in September 2025. Corroborated by 5+ independent outlets: [Noma Security (original research)](https://noma.security/blog/forcedleak-agent-risks-exposed-in-salesforce-agentforce/), [The Hacker News](https://thehackernews.com/2025/09/salesforce-patches-critical-forcedleak.html), [The Register](https://theregister.com/2025/09/26/salesforce_agentforce_forceleak_attack), [Security Affairs](https://securityaffairs.com/182676/hacking/forcedleak-flaw-in-salesforce-agentforce-exposes-crm-data-via-prompt-injection.html). Note: discovery predates our July-2026 horizon by about a year but remains actively cited through 2026 as the canonical agent-marketplace-adjacent trust failure — kept in scope as high-relevance, flagged for date.

### 1.4 Microsoft — Agent Store (Microsoft 365 Copilot / Copilot Studio)

- **Installed unit:** "agent," admitted via **four distinct pathways**, each with a different trust/approval path — the most structurally rigorous model in the survey. [Microsoft Learn — Agent Store](https://learn.microsoft.com/en-us/microsoft-365/copilot/copilot-agent-store) (primary/official source, fetched in full):
  1. **Prebuilt agents** from Microsoft/verified third-party providers — admin browses and deploys directly, no dev work.
  2. **Copilot Studio agents built in-org** — a maker builds and submits; an **admin must explicitly approve or reject** in the Microsoft 365 admin center before the agent is visible to any user; rejections route back to the maker with feedback.
  3. **External-platform agents** (Azure AI Foundry, custom code, other agent frameworks) — packaged via the Microsoft 365 Agents Toolkit/SDK, submitted to an org catalog, and likewise gated behind admin approval.
  4. **Agent 365 SDK integration** — agents get **Entra-backed identity**, governed access to M365 data via **managed MCP servers**, and inherit the org's compliance/governance "blueprint" automatically, appearing in a distinct "Agents for your team" collection.
- **Discovery UX:** Agent Store is **not a separate destination** — it's surfaced across Teams, Outlook, Word, Excel, and PowerPoint, i.e., embedded at the point of work rather than requiring a dedicated app-store visit.
- **Permission/trust model — explicit, UI-visible trust tiers:** each agent's detail pane exposes separate panels for **Publisher/Channel/Last updated**, **Users** (assign to just-me / entire org / specific groups), **Data & tools** (capabilities, knowledge sources, actions), **Security & compliance** (risks and protections), **Certification** (attestation info), and **Activity** (post-deployment usage data). Trust tiers are named and visually distinct in the catalog: "Made by Microsoft," "verified third-party providers," "your organization," and "Agent 365-enabled."
- **Scale at launch:** 70+ agents at launch with a "growing catalog" per Microsoft's own developer blog — official but **single-sourced** for the specific number.
- **Monetization:** not detailed in the sources reviewed; Microsoft's public materials focus on governance/adoption rather than partner revenue share for this specific surface.

### 1.5 Hugging Face — Spaces + Agentic Resource Discovery (ARD)

- **Installed unit:** Spaces itself predates the 2026 marketplace wave and is a **hosting/publishing layer**, not a curated store — largely open, community-driven, with no centralized pre-install vetting by default.
- **2026 development — ARD is a discovery protocol, not a marketplace.** It solves the "an agent can't hardcode every tool location, and dumping every tool description into context doesn't scale" problem via two mechanisms: (1) **static `ai-catalog.json` manifests** hosted by publishers at well-known URLs, and (2) a **dynamic `POST /search` registry API** returning ranked results carrying publisher identity, representative queries, compliance attestations, and tags. [HF ARD launch post](https://huggingface.co/blog/agentic-resource-discovery-launch) (fetched in full). ARD surfaces three resource types: MCP servers, Agent Skills, and other agents (via A2A).
- **Trust mechanism:** relies on **publisher-signed manifests and cryptographic verification** as a best practice, not a platform-enforced gate — community guidance explicitly says agents should "verify the publisher" and confirm resources "haven't been tampered with," but this is advisory, not mandatory.
- **HF's Discover Tool** wraps existing Spaces semantic search inside the ARD envelope — meaning Spaces doesn't become a new store; it becomes queryable inventory for any ARD-compliant agent, anywhere.
- **Cross-host portability:** HF-published Skills are explicitly built to work across **Claude Code, Codex, Gemini CLI, and Cursor** — the only player in this survey designing its package format to be host-agnostic from the start rather than locked to a single vendor's agent runtime. [huggingface/skills](https://github.com/huggingface/skills).

### 1.6 Google — Gemini Enterprise Agent Gallery, Workspace Marketplace, Gemini CLI Extensions

Google runs **three distinct trust regimes** under one brand, not one model:

- **Gemini Enterprise Agent Gallery (admin-gated):** agents grouped as Made-by-Google, organizational, custom (via Agent Designer), or Marketplace (via Google Cloud Marketplace + A2A protocol). Marketplace agents require a **user request → admin review → approval** flow before the agent appears in the user's "From your organization" section; admins also control whether the Marketplace section is even visible to end users. [Gemini Enterprise Agent Gallery docs](https://docs.cloud.google.com/gemini/enterprise/docs/agent-gallery) (fetched in full).
- **Google Workspace Marketplace (caveat-emptor):** extensions are sourced from public repositories by third-party developers; Google's own language states it **"does not vet, endorse, or guarantee the functionality or security of these extensions"** and tells users to inspect source code and requested permissions themselves before installing.
- **Gemini CLI extensions:** developer-facing, also sourced from public repos, similarly unvetted by Google — e.g., Hugging Face's own Gemini CLI extension is just one more public-repo listing. [Gemini CLI extensions](https://geminicli.com/extensions/).
- **Permission model where it exists:** standard **OAuth scope consent**, e.g. `cloud-platform`, `chat.messages.create`, `chat.spaces.create` plus Calendar/Gmail scopes as needed — scope-by-scope, the most granular *mechanism* in the survey, though it's a developer-facing OAuth screen rather than a curated trust signal for end users.
- **Net pattern:** Google explicitly bifurcates — strong admin gating for the enterprise agent surface, essentially no platform-level vetting for the open developer surfaces, with the vetting burden pushed onto the installing user/admin.

### 1.7 ClawHub / OpenClaw — the cautionary vertical case study

- **Model:** OpenClaw (formerly Clawdbot/Moltbot) is a self-hosted personal AI agent framework; **ClawHub** is its skill marketplace, explicitly modeled on **npm** — semver versioning, changelogs, update flows, and **vector/semantic search** discoverability. Scale: ~3,200 skills, 1.5M+ downloads — smallest of the majors, but the most "developer-native" packaging model in the survey. [Rapid Claw 2026 guide](https://rapidclaw.dev/blog/ai-agent-marketplace-guide-2026); [Digital Applied](https://www.digitalapplied.com/blog/ai-agent-marketplaces-2026-discovery-distribution).
- **Trust model at the time of incident: effectively none** — anyone could publish, and nothing was screened before listing.
- **The incident (corroborated by 5+ independent outlets, high confidence):** in **February 2026**, security firm **Koi Security** audited 2,857 ClawHub skills and found **341 malicious** (~12%), most tied to a single coordinated campaign dubbed **"ClawHavoc."** 335 of those disguised themselves as legitimate utilities (e.g., `solana-wallet-tracker`, `youtube-summarize-pro`, `better-polymarket`) while installing the **Atomic macOS Stealer (AMOS)** infostealer, which harvests browser credentials, keychain passwords, crypto wallets, SSH keys, and Telegram sessions. Other skills hid **reverse-shell backdoors inside otherwise-functional code**, or exfiltrated bot owners' own stored credentials (from local `.env` files) to attacker webhooks. A follow-up scan found the malicious count had **grown to 800+** across ~25 categories including browser-automation agents, coding assistants, and — notably — fake security-scanning skills. [The Hacker News](https://thehackernews.com/2026/02/researchers-find-341-malicious-clawhub.html); [Koi Security original research](https://www.koi.ai/blog/clawhavoc-341-malicious-clawedbot-skills-found-by-the-bot-they-were-targeting); [SC Media](https://www.scworld.com/news/openclaw-agents-targeted-with-341-malicious-clawhub-skills); [Unit 42 / Palo Alto](https://unit42.paloaltonetworks.com/openclaw-ai-supply-chain-risk/).
- **OWASP's own framing:** the OWASP Top 10 for Agentic Applications 2026 cites ClawHub as **"the first AI agent registry to be systematically poisoned at scale,"** noting 5 of the top 7 most-downloaded skills at peak infection were malware. [OWASP GenAI resource](https://genai.owasp.org/resource/owasp-top-10-for-agentic-applications-for-2026/).
- **Remediation, after the fact:** OpenClaw added skill vetting, runtime sandboxing, and a VirusTotal integration — reactive, not preventive.

### 1.8 Cross-cutting standard: OWASP Top 10 for Agentic Applications 2026

Announced at Black Hat Europe 2025 / the OWASP Agentic Security Summit, this gives the industry a shared vocabulary for marketplace-relevant failure modes (ASI01–ASI10): Agent Goal Hijack, Tool Misuse & Exploitation, **Agent Identity & Privilege Abuse**, **Agentic Supply Chain Compromise**, Unexpected Code Execution, Memory & Context Poisoning, Insecure Inter-Agent Communication, Cascading Agent Failures, Human-Agent Trust Exploitation, Rogue Agents. ForcedLeak maps cleanly to ASI03 (Identity & Privilege Abuse) and indirectly ASI01; ClawHub maps directly to ASI04 (Agentic Supply Chain Compromise). [OWASP GenAI](https://genai.owasp.org/resource/owasp-top-10-for-agentic-applications-for-2026/); [OWASP Agentic Skills Top 10](https://owasp.org/www-project-agentic-skills-top-10/).

## 3. What wins / what fails in these marketplaces

**What wins:**

1. **Discovery embedded at the point of work beats a separate destination.** Microsoft's Agent Store lives inside Teams/Outlook/Word rather than as its own app; Salesforce won adoption by *unifying* three separate marketplaces (AppExchange, Slack, Agentforce) into one rather than fragmenting further. A bolt-on "marketplace screen" is the weaker pattern.
2. **Visible, named trust tiers beat hidden vetting.** Microsoft's "Made by Microsoft / verified third-party / your organization / Agent 365-enabled" labels let a user calibrate trust at a glance. Salesforce's and Anthropic's review processes exist but aren't surfaced as clearly in the install UI itself.
3. **Protocol-level neutrality builds durable multi-vendor trust.** Anthropic's decision to hand MCP to the Linux Foundation-backed Agentic AI Foundation (with OpenAI, Block, Google, Microsoft, AWS as co-sponsors) is the only example in this survey of a single vendor deliberately un-coupling governance from its own commercial interest — a credibility move none of the storefront-style marketplaces (Salesforce, Microsoft, Google, OpenAI) have made for their own catalogs.
4. **Host-agnostic package formats grow supply faster.** Hugging Face's Skills format works across Claude Code, Codex, Gemini CLI, and Cursor — the only publisher-side format in the survey not locked to one vendor's runtime, which lowers the cost of a publisher supporting it.
5. **Conservative, staged monetization avoids solving trust and payments at once.** OpenAI's checkout tiering (external → saved-method → payment-sheet-beta-for-vetted-partners-only) lets commerce trust mature gradually instead of exposing every developer to payment risk on day one.

**What fails:**

1. **Install-time-only vetting, with no runtime/behavioral layer, is a proven failure mode.** ForcedLeak succeeded against an agent that had passed Salesforce's install-time security review — the exploit was a runtime indirect-prompt-injection plus a stale CSP allowlist entry, neither of which a one-time review catches. A named critic of the AgentExchange launch put it plainly: liability and behavioral drift "are not in the press release."
2. **No/self-serve publishing at scale is catastrophic without pre-screening.** ClawHub's npm-style, anyone-can-publish, semantic-search-discoverable model produced a ~12% malicious-skill rate before intervention and real credential/crypto theft — the single worst outcome documented in this survey, and OWASP now cites it as the reference case for "Agentic Supply Chain Compromise."
3. **Explicit non-vetting disclaimers push real security work onto end users who have neither the tooling nor the incentive to do it.** Google's Workspace Marketplace and Gemini CLI extensions both carry "we don't vet this, inspect the code yourself" language — technically honest, but it likely under-reports real incident rates since most users never actually inspect source before installing.
4. **Category-noun sprawl slows trust-building industry-wide.** No player has one clean term — "app," "agent," "skill," "connector," "tool," "extension," "workflow" are all in simultaneous use, often for overlapping concepts, even within a single vendor's own ecosystem (Anthropic alone runs MCP servers, Skills, Connectors, and Claude Code plugins as four separate nouns).
5. **Overpromising in-agent commerce ambition gets walked back even at the best-resourced player.** OpenAI's partial retreat from full in-chat checkout to merchant-storefront redirects shows that commerce liability inside an agent surface is still an unsolved problem in mid-2026, not a solved-and-scaling one.

## 4. Where this contradicts or outpaces the Sonik baseline

Sonik's baseline: one package envelope with first-class `kind` (bundle/app/workflow/skill/command_tool_pack/agent/artifact_template/mcp_addon/provider_integration); installs target immutable `packageVersionId`; JSON-render apps are execution-inert (renderer collects input only, a trusted controller commits via host-signed approval); permission grants are `off/ask/allow`.

1. **Sonik's unified kind taxonomy is genuinely ahead of the field — the industry has category-noun sprawl, Sonik doesn't.** No reference player has one clean packaging vocabulary; Anthropic alone runs four separate nouns for conceptually similar things. This is a real, citable advantage worth stating with confidence, not just defending.
2. **But the field's best player (Microsoft) wins by surfacing named, user-visible trust tiers separately from the permission-grant mechanism — Sonik's `off/ask/allow` doesn't obviously do this.** `off/ask/allow` is a runtime grant, not a publisher-provenance label. Microsoft's "Made by Microsoft / verified partner / your org / Agent-365" labels are shown *in the catalog itself*, before a user ever reaches a permission prompt. **Delta: Sonik should consider a first-class, user-visible provenance/trust-tier attribute per package (e.g., first-party / verified-partner / community), independent of the kind taxonomy and independent of the off/ask/allow grant**, since discovery-time trust signaling and runtime permission grants are solving different problems in the leading implementation.
3. **Install-time vetting frozen at an immutable `packageVersionId` is exactly the pattern that failed for Salesforce (ForcedLeak) and catastrophically for ClawHub.** Immutability is good for reproducibility, but neither incident was caused by a *bad version being installed* — both were runtime exploitation of an already-approved, already-installed capability (indirect prompt injection in one case, malicious code shipped as a "legitimate" skill in the other). **Delta: pair immutable `packageVersionId` installs with a post-install monitoring/revocation lever** — the ability to kill or quarantine a specific installed `packageVersionId` fleet-wide when new threat intel emerges, and ideally a Microsoft-style per-install "Activity" view — rather than treating install-time approval as a one-time, permanent trust decision.
4. **Sonik's execution-inert JSON-render + trusted-controller-commit + host-signed-approval design is structurally safer than most of this survey and should be treated as a competitive claim, not just an internal default.** It's closer in spirit to OpenAI's sandboxed-iframe-plus-explicit-tool-call-approval than to ClawHub's arbitrary-code-execution skills or Hugging Face's largely unenforced Spaces trust model. Worth stating explicitly in positioning: most category-defining marketplaces in mid-2026 either allow arbitrary code execution on install (ClawHub) or rely on advisory/community trust signals (HF/ARD) rather than a structural execution-inert-by-default rendering model.
5. **No reference marketplace treats supply-chain governance as a protocol-neutral question the way MCP's Linux-Foundation donation does — this is a strategic choice Sonik should make deliberately, not by default.** If the package envelope (amp.pkg) is ever meant to be portable beyond Sonik's own host, or opened to third-party publishers at scale, the MCP donation is a signal that credible multi-vendor ecosystems eventually decouple governance from the single vendor running the marketplace. Every storefront-style marketplace in this survey (Salesforce, Microsoft, Google, OpenAI) instead keeps governance in-house — which is a perfectly valid choice for a single-host, first-party marketplace, but Sonik should pick one path explicitly rather than drift into it.
6. **A flat three-state `off/ask/allow` grant is coarser than what every enterprise-grade competitor exposes at scale.** Microsoft separates Users/Data & tools/Security & compliance/Activity into distinct panels; Salesforce controls at user/role/profile granularity; Google uses full OAuth scope lists. **Delta: as `provider_integration` and `mcp_addon` kinds proliferate, `off/ask/allow` likely needs a fourth axis — scope** (which specific data/API boundary a grant covers) — not just a binary-ish gate on the whole package.
7. **Discovery placement is a real gap risk, not addressed by the baseline as described.** The two strongest adoption stories in this survey (Microsoft, Salesforce) both win by embedding discovery into the surfaces people already use rather than treating the marketplace as a destination. If Sonik's marketplace today is a standalone screen inside the booking platform rather than surfaced contextually at the moment a booking/ops task could use a workflow or skill, that's a discovery-UX gap versus the category leaders — worth flagging even though it's outside the package/permission model itself.

---

## 5. So-what for Sonik

- **Keep and lean into the unified package-envelope/kind taxonomy** — it's a genuine, citable advantage over an industry still fragmented across app/agent/skill/connector/tool nouns (Anthropic alone runs four).
- **Add a user-visible provenance/trust-tier label per package** (first-party / verified-partner / community), separate from `kind` and separate from the `off/ask/allow` grant — the category leader (Microsoft) wins specifically by surfacing this at discovery time, not burying it in permission plumbing.
- **Pair immutable `packageVersionId` installs with a post-install kill-switch/monitoring lever.** Both of this survey's worst incidents (ForcedLeak's CVSS 9.4 runtime exploit against an already-reviewed Salesforce agent, and ClawHub's ~12%-malicious skill rate) were runtime failures of already-approved capabilities, not bad-version installs — install-time review alone is proven insufficient.
- **Position execution-inert JSON-render apps + host-signed trusted-controller commits as a competitive safety claim**, not just an internal default — it's structurally stronger than ClawHub's arbitrary-code-execution model and Hugging Face's advisory-only trust model, and comparable to OpenAI's sandboxed-iframe approach.
- **Expand `off/ask/allow` with an explicit scope dimension** (what specific data/API boundary each grant covers) before `provider_integration`/`mcp_addon` kinds proliferate — every enterprise-grade competitor in this survey (Microsoft, Salesforce, Google) already exposes finer-grained control than a flat three-state gate.
