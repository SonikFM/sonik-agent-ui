# Lane L2 — Workflow/Template Marketplace Packaging & Install UX (2026-07)

Status: DRAFT · Research complete 2026-07-06
Lane: L2 of the agent-marketplace research plan (`agent-marketplace-research-plan-2026-07-06.md`)
Method: `deep-research` skill — 5 parallel search lanes (n8n; Make + Zapier; Notion + Airtable + Gumroad; Retool + Dify; 2025-2026 AI agent marketplace entrants), each independently sourced and cross-checked. Search horizon: 2025-2026, pre-2025 sources flagged inline as stale/context-only.

Confidence key: **[2+]** = corroborated across 2+ independent sources. **[single-source]** = only one source found; treat as directional, not settled fact.

## 1. Per-product packaging & install-UX profiles

### n8n

- **Packaging unit**: a workflow JSON file — `nodes` (id, name, type, typeVersion, parameters, position), `connections` (keyed graph of node outputs→inputs), `settings`. Credentials are referenced by name/ID, never embedded as secrets. **[2+]** (docs.n8n.io/build/manage-workflows/export-and-import, docs.n8n.io/courses/level-one/chapter-6)
- **Discovery**: n8n.io/workflows, a public marketplace-style gallery (~10,400 templates, ~7,200 in the AI category alone at time of research), browsable by category or by "what's in your stack," plus a "Verified Creators" directory.
- **Install/fork**: templates open directly inside the user's own instance (cloud or self-hosted) via a "Use for free" CTA, or via manual Import from File/URL. **No automatic credential remapping** — the user must manually reassign every credential reference after import; this is explicit in n8n's own docs and is common enough that community members have built dedicated "map credentials on import" template workarounds. **[2+]**
- **Fork independence**: import creates an immediately independent copy with no live link back to the source template; there is no way to tell, from inside an imported workflow, which upstream template version it came from. **[medium]** (inferred from absence of any contrary documentation, corroborated by a community thread on the topic)
- **Versioning**: templates themselves carry no consumer-visible version number, and template updates do **not** propagate to prior imports — each import is a frozen snapshot. n8n's own "workflow history" feature versions a user's *post-import edits*, not the upstream template, and its retention is tiered (24h free / 5 days Cloud Pro / full on Enterprise).
- **2025-2026 shift**: AI-agent templates now reportedly dominate new submissions (~70-80%, **[single-source, third-party aggregator]**); self-hosted instances can white-label their template source via `N8N_TEMPLATES_HOST`; n8n added an MCP Server so external AI systems can call n8n workflows directly.
- **Monetization**: a "Verified Creators" program exists (directory + community perks) but **no confirmed revenue-share or payout mechanism** from n8n itself — creator-earnings claims in circulation are unverified marketing content.

### Make (formerly Integromat)

- **Packaging unit**: a "blueprint" JSON — `name`, `flow` (module objects with per-module version), `metadata` (scenario settings + a top-level blueprint-format version, currently `1`). Blueprints do **not** carry live connections/credentials. **[2+]** (help.make.com/blueprints, developers.make.com scenario/blueprint API, corroborated by a community blueprint-structure reference)
- **Discovery**: make.com/en/templates, filterable by app or category, two classes of templates — "public" (community/Make-authored) and "team" (private, org-shared).
- **Install/fork**: "Create from template" copies the blueprint into a new scenario in the user's own account; every module's auth/connection must be set up manually post-copy. An optional guided-setup wizard (checkmark/star/exclamation completeness indicators) softens this. **[2+]**
- **Versioning**: **no formal template versioning or auto-update mechanism**. When a creator republishes an updated template, users who already created a scenario from an earlier version are simply unaffected — no notification, no sync. **[single-source, but from Make's own help center]**
- **Monetization**: no template-specific creator program. Make's three partner tracks (Affiliate 35% commission/24mo, Partner Program for service providers, ISV/Technology Partner for connectors) are unrelated to template authorship.

### Zapier

- **Packaging unit (classic Zap template)**: trigger + action steps with pre-mapped fields; Zapier's own authoring guidance tells template builders to leave user-specific fields (e.g., folder/project IDs) blank rather than defaulting them.
- **Important correction to framing**: "Zapier Central" no longer exists as a separate product — it launched non-agentically in March 2024 and was folded into **Zapier Agents** in January 2025. **[2+]** (Zapier community post, MIT AI Agent Index). Agents are packaged with attached "knowledge sources" (Drive, Notion, Confluence, uploads) rather than a fixed trigger/action chain.
- **Install/fork — classic**: "Try it"/"Use this Zap" opens a pre-filled editor; connect accounts, map remaining fields, publish. A newer **"Guided templates"** feature turns a fully-configured Zap into a step-by-step wizard where the creator can lock/unlock individual fields and attach per-step help text — a deliberate lowering of the reconfiguration burden for non-technical recipients.
- **Install/fork — Agents**: sharing generates a template link; recipient sees pre-populated settings, edits, clicks "Use this template" — result is an **independent copy**, no live link back to the source agent.
- **Versioning — classic templates**: no explicit version numbers; templates inherit the underlying app-integration's version. Non-breaking API changes auto-propagate; breaking changes leave old templates untouched; deprecating an app version marks dependent templates "invalid" within 24 hours. Notably, **template authors cannot self-edit a published template** — they must file a support request to revert it to draft. **[2+]**
- **Versioning — Agents**: real draft/publish versioning (v1, v2, v3…) — edits happen in a draft; the live agent is untouched until explicitly published. Zapier Copilot (AI-assisted Zap building) added a "checkpoint" diff in 2026 showing exactly what it added/removed/rewrote on each edit. **[single-source for the checkpoint detail]**
- **Discovery-to-first-value**: classic templates aim for near-zero friction — browse → "Use this Zap" → connect 1-2 accounts → publish, roughly 3-5 interactions by design. Agent templates run browse (zapier.com/agents/templates, ~100 curated) → "Use this template" → configure unlocked fields → activate; no precise benchmark was found for Agent flow specifically.

### Notion (Template Gallery / Marketplace)

- **Packaging unit**: a full page-tree duplication — the template, all subpages, and embedded databases copy into the user's sidebar. Whether formulas and cross-workspace linked databases fully survive duplication is **under-documented** by Notion itself.
- **Structural change**: Notion rebranded its Template Gallery into a unified **Marketplace** (30,000+ templates) — one surface now for both browsing and selling, with creator profiles, Stripe onboarding, and a seller dashboard. **[2+]**
- **Install/fork**: "Duplicate" creates an immediately independent copy with no live sync back to source. There is **no official update-propagation mechanism** — the sanctioned pattern for a creator pushing an update is: buyer manually re-duplicates from the original link into a fresh copy, then migrates their data over by hand. **[single-source, but consistent with the stateless-duplicate model found across all sources]**
- **Monetization**: confirmed and detailed — Notion is merchant of record, takes 10% + $0.40/transaction (+1% FX for international), handles tax/VAT, pays out biweekly with a $20 minimum and a 14-day refund-risk hold. **[2+]** Creators commonly multi-home on Gumroad/own storefronts specifically to avoid this cut over time.

### Airtable Universe

- **Packaging unit**: base + tables + views + data; interfaces travel with the base. **Automations are copy-path-dependent**: a true in-workspace "Duplicate base" copies automations (disabled by default), but copying via a public share link or from Universe **never copies automations** — confirmed by an Airtable community moderator. **[flagged as possibly stale — sourced from a 2023 thread, mechanic not independently re-verified for 2026]**
- **Install/fork**: "Use template" → choose workspace → "Add base." Because external API keys/sync connections and (via Universe) automations are never carried over, this is a **meaningfully heavier reconfiguration burden** than Notion's duplicate flow.
- **Versioning**: snapshot-only, no update mechanism found for previously copied bases.
- **Status flag**: Airtable's support docs (last updated May 2026 at fetch time) still present Universe as active, but community-manager responses describe it as **soft-deprecated** — new submissions aren't indexed even though existing links still work, with community support steering toward "official templates" and a "Show and Tell" board instead. **[single-source, but from an Airtable community manager]**
- **Monetization**: none. Free community-sharing feature, not a two-sided marketplace.

### Gumroad (generic creator-economy comparison point)

- Not an in-product gallery — a generic storefront for arbitrary digital files (ZIPs, license-gated software, PDFs). No "packaging unit" concept: no live app state is duplicated, only static file/license delivery.
- Fees: 10% + ~2.9%+$0.30 processing on direct/profile sales, jumping to **30%** for sales sourced via Gumroad's internal "Discover" surface. **[2+]**
- Key structural contrast: in-product galleries (Notion, Airtable, n8n, Make) duplicate *live, structured application state*; Gumroad only ships static files with no runtime semantics.

### Retool

- **Packaging unit**: apps are stored as structured JSON (layout, components, queries, logic). Reusable sub-units ship as **Modules** — a package of components + queries/variables/transformers, exportable/importable as JSON, where an update to a Module **does propagate live** to every parent app that references it. This is the closest analogue in the whole survey to true "template update propagation" — but it only works within one org's own module library, not across public template installs.
- **Discovery**: a flat gallery at retool.com/templates by use case; no distinct community-template hub — community examples surface on the forum and third-party blogs instead.
- **Install/fork**: "Try this template" spins up an org with the template pre-loaded, but datasource credentials are never bundled — Retool's Resources model deliberately decouples credentials (DB/API/OAuth) from the app JSON; queries bind to a Resource by reference, and the user must create/connect their own Resource and rebind after import.
- **Versioning**: two real systems — **Releases** (semantic-version snapshots, draft vs. published live release, Team/Business/Enterprise) and **Source Control** (git-backed branches/commits/PRs against GitHub/GitLab/CodeCommit, Enterprise-oriented, with a multi-instance release manifest for dev→staging→prod promotion). Neither system documents whether marketplace templates themselves carry version metadata or push updates to already-installed copies — an **unaddressed gap** in Retool's public docs.
- **2025-2026 AI packaging**: a dedicated **Retool Agents** product — build from a template or scratch, attach an LLM, give access to saved queries/workflows, link MCP servers or other agents.
- **Monetization**: none found; a 2019 community feature request for a template marketplace appears to have never shipped.

### Dify

- **Packaging unit**: a DSL in YAML (app config/metadata, workflow orchestration/node settings, model params, prompt templates, knowledge-base *references* not KB content itself). Third-party API keys, KB content, logs, and secret env vars are excluded from export by default (opt-in per field). **[2+]**
- **Discovery**: two surfaces — an in-app "Explore" section for one-click adoptable pre-built apps, and a newer standalone **marketplace.dify.ai** with a "Templates" tab (distinct from the existing Plugins tab), filterable by category/tag/creator, with a graph-view workflow preview before install. This Template Marketplace + Creator Center is a genuine **2025-2026 addition**, not legacy.
- **Install/fork**: upload a DSL YAML; a version check warns if the DSL predates the current platform but the platform always runs the latest DSL version, so imports are always compatible. From the marketplace, "Open in Dify" + guided configuration. Exact field-level copy-vs-reference behavior for KB linkage is **not fully documented** — a genuine gap, not a confirmed mechanism.
- **Versioning**: a **Version Control Panel** for Chatflow/Workflow apps — publish named versions with release notes, chronological history, restore any prior published version into the current draft. **[single confirmed source, moderate confidence]**. Whether marketplace *template* updates propagate to already-installed copies is undocumented; DSL import behaves as a one-time snapshot with no live link back to source, unlike Retool's live-propagating Modules.
- **2025-2026 plugin ecosystem**: Dify v1.0.0 (Feb 2025) decoupled models/tools into a Plugin system (Models, Tools, Data Sources, Triggers, Agent Strategy, Extensions, Bundles), each code-reviewed and run in permission-isolated sandboxes. **[2+]**
- **Monetization**: confirmed — a Creator affiliate program via PartnerStack, **up to 50% recurring commission** when a user subscribes/pays through a creator's template link. **[single-source, but first-party/primary — Dify's own announcement]**. This is the most generous, most explicit creator-revenue model found in the entire survey.

### 2025-2026 new entrants and emerging standards

- **Relay.app**: one-click shareable template link → independent copy in recipient's workspace; no public marketplace directory, point-to-point/team-library sharing only.
- **Gumloop**: public community library (45+ flows) plus a **paid marketplace in development** where buyers clone templates into their workspace for a few dollars — clone/copy model, not subscription-to-source. **[single-source on the paid tier]**
- **CrewAI Marketplace**: templates packaged as full GitHub scaffold repos (`pyproject.toml`/UV deps, `agents.yaml`/`tasks.yaml`), installed via `crewai install`; approved templates get hosted *inside* CrewAI Enterprise (referenced, not just copied); a revenue-share program is planned but pre-launch as of research date.
- **LangGraph Templates**: full clonable GitHub repos — explicitly "clone the repo, own all the code," versioned on plain git rather than a central registry; launched with only 4 templates, expanding gradually (LangChain/LangGraph hit v1.0 in Oct 2025).
- **OpenAI Apps SDK / ChatGPT App Directory** — the single most consequential 2025-2026 launch in this space. Preview Nov 13 2025, directory + developer submission opened Dec 18 2025, reaching 800M+ ChatGPT users. Packaging unit is an **MCP server** plus an optional sandboxed-iframe web UI — Apps SDK is built on top of MCP, not a bespoke format. "Install" means *enabling* an app from the directory — a live connected integration, not a copied/forked artifact. **[2+]**
- **Anthropic Claude Skills**: SKILL.md-based packages (instructions/scripts/resources folder), official marketplace (`anthropics/claude-plugins-official`) plus a community marketplace (`anthropics/claude-plugins-community`) gated by automated validation/safety screening.
- **MCP Registry** (official, preview since Sept 2025): a `server.json` manifest that *points to* existing package registries (npm/PyPI/Docker Hub) rather than bundling code — namespace ownership enforced via reverse-DNS names tied to verified GitHub or domain-verification. Still pre-1.0 (v0.1 API freeze). Third-party aggregators (Smithery ~7,000 servers with its own `smithery.yaml`+Docker packaging; MCPMarket.com 10,000+ servers) sit on top.
- **A2A (Agent2Agent) protocol**, launched by Google April 2025: Agent Cards advertise capability metadata, Tasks structure exchanged work with lifecycle states, transport is HTTP/SSE/JSON-RPC 2.0. 150+ backing orgs by April 2026 (Google, Microsoft, AWS, Salesforce, SAP, ServiceNow, Workday, IBM). **[2+]**
- **Microsoft Copilot Studio Agent Library**, **Salesforce AgentExchange** (renamed from AppExchange at TDX 2026, 200+ launch partners, agent templates bundling topics + partner actions + global instructions), **ServiceNow AI Agent Marketplace** (overhauled partner tiers Jan 2026) — all enterprise vendors converged on an "agent template marketplace" in the same ~12-month window.
- **Failure mode, most significant finding of this lane**: a real 2026 supply-chain security incident on **ClawHub**, a third-party skill marketplace for the "OpenClaw" agent framework. Koi Security's audit found 341 of 2,857 reviewed skills malicious (credential-stealing malware, campaign "ClawHavoc"); follow-up scans found 800+ malicious skills as the marketplace grew; Antiy CERT separately confirmed 1,184 malicious skills. Five malicious skills evaded the marketplace's own VirusTotal + "ClawScan" review layer via base64 encoding/file padding between Feb-May 2026. Corroborated by two independent security outlets (Unit 42/Palo Alto Networks, Dark Reading). **[2+], directly relevant to trust/supply-chain concerns for any marketplace accepting user-authored executable/instruction packages.**

## 3. Best-in-class install→first-value patterns

Ranked by how little post-install reconfiguration stands between "discover" and "working":

1. **OpenAI Apps SDK / ChatGPT Apps — near-zero friction, but because it isn't a fork at all.** Because the packaging unit is a live MCP-backed connection rather than a copied config, "install" is just "enable." This sidesteps the entire credential-remapping problem every fork-based tool (n8n, Make, Zapier, Retool) has to solve — at the cost of the user never owning an editable copy. This is the sharpest structural alternative to "copy-and-own" in the whole survey.
2. **Zapier's Guided Templates** — the best answer found to the fork-then-reconfigure problem *without* giving up ownership. The template author pre-locks fields that must not change and attaches inline help text per remaining field, turning "here's a JSON dump, go fix it" into a directed wizard. This is a UX layer bolted onto the same underlying copy-and-fork packaging model everyone else uses — it doesn't change the unit installed, just the reconfiguration experience.
3. **Retool Modules' live-propagation** — the only mechanism in the survey where an installed/referenced unit actually receives upstream updates automatically, but it's scoped to a single org's internal module library, not to public template marketplace installs. No vendor in this survey has solved "push an update to a stranger's already-forked copy" at the public marketplace layer — n8n, Make, Zapier, Notion, Airtable, and Dify are all snapshot-at-import with no propagation, by design or by gap.
4. **Notion Marketplace's monetization plumbing** (Stripe-based payout, tax handling, refund-risk holds) is the most mature creator-monetization implementation found, but its actual duplicate/update mechanics are the weakest in the survey — no formal versioning, no update path beyond "re-duplicate and manually migrate your data." Good business-model UX, poor artifact-lifecycle UX.
5. **Dify's Creator Center (50% recurring commission)** pairs a real technical versioning system (Version Control Panel with rollback) with the most generous monetization terms found — the strongest combined package/monetization story among tools with an actual "your own copy, in your own environment" model (as opposed to OpenAI's live-connection model).

## 4. Where this contradicts or outpaces the Sonik baseline

The signed-off baseline (per the research plan) is: one package envelope, first-class kinds (`bundle/app/workflow/skill/command_tool_pack/agent/artifact_template/mcp_addon/provider_integration`), installs target an immutable `packageVersionId`, JSON-render is canonical with HTML as escape hatch, trusted-host context gates write/destructive/external effects, bundles are default composition with individual apps/workflows/skills first-class.

- **Immutable `packageVersionId` is stronger than almost everything in this survey, and that's a genuine differentiator worth keeping — not weakening it.** Every fork-based competitor (n8n, Make, Zapier, Notion, Airtable, Dify) ships templates with **no consumer-visible version identity at all** — "version" describes the underlying platform/blueprint format, not the template artifact itself, and update propagation to already-installed copies is uniformly unsolved. Sonik's model of an immutable, addressable `packageVersionId` is *ahead* of the market here, not behind it. The market gap is real: nobody has cracked "notify/update a stranger's forked copy," so if Sonik wants to differentiate, this is an open lane, not a caught-up-to-standard.
- **Where Sonik is heavier than the market: credential/connection re-binding on install.** Every competitor that forks a real artifact (n8n, Make, Zapier, Retool) treats credential rebinding as an unavoidable, mostly-manual step after copy — this is normal, not something Sonik needs to solve better than the field. But it's also not solved well anywhere; Zapier's Guided Templates (locked fields + per-step help text) is the best UX mitigation found and is directly portable to Sonik's install flow for workflow/bundle packages that require host-scoped credentials or provider connections.
- **Where the market is outpacing a copy-based model entirely: the OpenAI Apps SDK "enable, don't copy" pattern.** If Sonik's read of "installable capability for an embedded agent" leans toward capabilities that should stay live-connected to a provider/service (vs. a workflow graph a customer edits), the emerging reference model isn't fork-and-own, it's MCP-backed live connection with directory-level enable/disable. This directly informs the `provider_integration`/`mcp_addon` package kinds in the baseline — those categories should probably NOT behave like `workflow`/`artifact_template` (copy-and-own); they should behave more like the Apps SDK model (reference a live, versioned server-side capability, no local fork). Worth checking whether the current baseline treats all package kinds with one uniform install semantic or differentiates by kind.
- **Where the market outpaces Sonik on trust/supply-chain given user-authored executable content: none of the mainstream template galleries (n8n/Make/Zapier/Notion/Airtable) have a serious review pipeline for malicious content because their templates are largely declarative config, not arbitrary code/instructions.** But the ClawHub/OpenClaw incident (341-1,184+ malicious skills found across multiple audits, some evading automated review) is a direct, concrete warning for any marketplace — including Sonik's — that accepts user- or third-party-authored `skill`/`command_tool_pack`/`agent` packages with real execution privilege. Sonik's "trusted host context required for write/destructive/external effects" gate is directionally the right defense, but this incident shows automated scanning (VirusTotal-style + custom scanners) alone is insufficiently robust against encoding/padding evasion — human review and/or capability-scoped sandboxing matter more than scanning at the marketplace-ingestion layer.
- **Bundles-as-default composition is a genuine point of difference vs. every competitor surveyed here.** Every product in this lane — n8n, Make, Zapier, Notion, Airtable, Retool, Dify — installs a *single* artifact per action (one workflow, one base, one app). None model a first-class "bundle of related apps/workflows/skills" as the default installable unit the way Sonik's baseline does. This isn't a place Sonik is behind; it's an area where there's no market precedent to validate against one way or the other from this lane — L1 (agent/AI marketplace lane) or L5 (vertical/wow-UX lane) may have better comparables (e.g., app-suite bundles in vertical SaaS marketplaces).

## 5. So-what bullets for Sonik

1. **Keep and market the immutable `packageVersionId` model as a real differentiator** — no competitor in workflow/template marketplaces gives installed copies a durable version identity or a credible update path; Sonik already has the harder half of a problem the entire market has left unsolved.
2. **Build a Zapier-Guided-Templates-style locked-field/wizard layer for install-time credential and config binding** — this is the single most transferable UX pattern found, and it directly addresses the reconfiguration friction that every fork-based competitor (and Sonik's own workflow/artifact_template installs) inherits by nature of decoupling credentials from package JSON.
3. **Differentiate install semantics by package kind rather than applying one uniform "copy-and-own" model everywhere** — treat `provider_integration`/`mcp_addon` more like the OpenAI Apps SDK's "enable a live connection" pattern, and reserve fork-and-own semantics for `workflow`/`artifact_template`/`bundle` kinds where a customer genuinely wants an editable copy.
4. **Treat the ClawHub/OpenClaw incident as a forcing function for review depth, not just automated scanning, on any package kind with execution privilege** (`skill`, `command_tool_pack`, `agent`) — automated content scanners were bypassed via simple encoding tricks in a comparable marketplace within the last few months; Sonik's trusted-host-context gate should be paired with meaningful human/curated review before opening third-party publishing.
5. **There is no validated precedent yet for "bundle as default installable unit"** — none of the workflow/template marketplaces surveyed model bundles this way, so this piece of the baseline should be validated against L1 (agent marketplace reference implementations) and L5 (vertical SaaS / wow-UX), not assumed correct on the strength of this lane alone.
