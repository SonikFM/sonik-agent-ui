# Lane L3: Permission, Trust, and Supply-Chain Models for Install-Time Review of Executable Marketplace Content

**Prepared:** July 2026 (search horizon 2025–2026; older sources flagged where used as historical baseline)
**Scope:** How the best software ecosystems keep users safe when installing things that can act on their behalf — and what Sonik's agent-marketplace design should take from them.

**Confidence key:** ✅ = corroborated by 2+ independent sources (or a primary spec/vendor doc directly fetched and confirmed). ⚠️ = single-source or vendor-blog-only; treat as directional, verify before quoting externally.

---

## 1. How each ecosystem handles install-time trust

No mainstream ecosystem has solved this problem cleanly. Every one of them makes a different trade-off between friction and safety, and every one of them has been breached through its own trust mechanism at least once in the 2025–2026 window. The honest summary: **install-time trust signals are point-in-time attestations, not continuous guarantees** — a theme confirmed by both independent security research (OX Security on VS Code) and Anthropic's own engineering writing (on remote MCP tools). ✅

| Ecosystem | Pre-publish gate | Install-time UX | Granular capability control | Re-consent on update |
|---|---|---|---|---|
| **VS Code Marketplace** | Automated malware scan (Defender-derived) + signature; no human review | Publisher-trust dialog only — no capability list, because the extension host runs with full VS Code privileges | None — all-or-nothing; Workspace Trust gates *workspace* code, not the *extension itself* | Not found to exist — extensions auto-update with no re-prompt |
| **Open VSX** (Cursor/Windsurf/Theia backend) | Reactive — automated scanning only added *after* Oct 2025 GlassWorm incident | Same all-or-nothing model as VS Code | None | None found |
| **Chrome Web Store (MV3)** | Automated + policy review; remote code execution banned in MV3 | Permission list shown at install (`host_permissions`, declared APIs); Enhanced Safe Browsing flags untrusted developers | Partial — `optional_permissions` allow contextual, in-feature requests | ✅ Yes — extension is auto-disabled if an update adds a permission producing a new warning message, until user re-accepts |
| **Firefox/AMO** | Automated **+ manual** code review with functional testing (more human-review-heavy than Chrome or VS Code) | Standardized "Extension Data Consent" screen (May 2025) shown at install | Policy requires minimum-necessary permissions | Not explicitly confirmed in this pass |
| **npm / PyPI** | Automated malware/behavior scanning (npm, 2025 initiative); PyPI Project Quarantine (Dec 2024) | None at "install" — trust is implicit in `npm install`/`pip install`; provenance/attestation badges visible on registry page if publisher opted in | None — a package either runs its install scripts or doesn't | N/A (packages aren't "updated" with re-consent; lockfiles pin versions) |
| **Google OAuth** | CASA tiered security assessment (self-assessment → 3rd-party DAST → pentest) for sensitive/restricted scopes | Granular per-scope checkboxes (partial-grant supported) since 2019; incremental authorization requests scopes only when a feature is used | ✅ Yes — closest real-world granular consent model | ✅ Adding new sensitive/restricted scopes forces re-verification before the app can call them |
| **Apple App Store** | Automated scan + ~300 human reviewers; Privacy Nutrition Labels self-declared | Runtime contextual prompts (since iOS 6, 2012), one-shot with usage-purpose string | Per-capability runtime toggles in Settings; per-contact sharing added iOS 18 | Contextual re-prompt only when new code path executes; no bulk re-consent on update |
| **Google Play** | Play Protect scan + review; Play Integrity API (successor to SafetyNet) attests device/app state | Runtime dangerous-permission prompts (Android 6+, 2015); Data Safety label at listing | Per-permission runtime toggles | Contextual, not bulk |
| **Slack / Microsoft Teams** | App Directory submission review | Full OAuth v2 scope list at install (Slack); Resource-Specific Consent, team-scoped (Teams) | Teams RSC is genuinely granular per-team/per-resource | Not confirmed for Teams; Slack requires reconnection on scope change |
| **OpenAI GPT Store / Apps** | Identity/business verification + automated + manual policy review | — | **Action control**: allow-all / read-only / custom action set — the most granular *admin* toggle found outside Anthropic's own products ⚠️ (single-sourced to OpenAI help center) | Not confirmed |
| **Anthropic Claude.ai / Claude Code** | Three-tier: official (Anthropic-curated) → community (automated validation + safety screening, pinned to commit SHA) → fully unreviewed self-hosted (add by URL/git repo) | Explicit disclaimer at install: *"Anthropic doesn't control what MCP servers, files, or other software are included in plugins and can't verify that they work as intended"* ✅ (directly fetched, confirmed verbatim) | ✅ Genuine off/ask/allow-style per-action permission (always-allow / needs-approval / block) — but this is **post-install runtime configuration**, not a single install-time gate | Not applicable in the traditional sense — Anthropic's own engineering post states remote MCP tools "can change behavior at any point after you've approved" them, i.e., there is no re-consent trigger tied to a manifest diff |
| **Salesforce AppExchange** | Paid security review (~$999, 4–5 weeks) ⚠️ | — | — | Uninstalled connected apps are blocked by default and require admin re-approval to reactivate ⚠️ (defensive-by-default, unusual and worth copying) |
| **WordPress plugin directory** | Essentially none | — | None | None — 97% of WordPress CVEs originate in plugins/themes (Patchstack 2025), the predictable result of weak gating |

**Verified directly from primary sources (not secondhand):**
- Anthropic's Claude Code docs, fetched directly: plugins are "highly trusted components that can execute arbitrary code on your machine with your user privileges," and the community marketplace's screening is "automated validation and safety screening" — explicitly *not* human security review, and explicitly *not* a behavioral guarantee. ✅
- Anthropic's "How we contain Claude" engineering post (May 25, 2026, fetched directly): Claude Code's manual-approval mode saw ~93% blanket approval rates ("approval fatigue"), which is why Anthropic moved to OS-level sandboxing (Seatbelt/bubblewrap — 84% reduction in prompts needed) and a probabilistic auto-mode classifier (~83% catch rate on risky actions, meaning ~17% of risky actions still get through even with the classifier). This is a direct, quantified illustration of why per-action approval prompts degrade to rubber-stamping at scale — directly relevant to any off/ask/allow design Sonik builds. ✅
- The MCP authorization spec, fetched directly: authorization is explicitly **OPTIONAL**, and **STDIO transport implementations "SHOULD NOT" follow the authorization spec at all**, instead pulling credentials straight from the environment. ✅ This means a large share of real-world MCP servers (anything running as a local subprocess, which is the majority of developer-tool MCP servers today) has no protocol-level authorization boundary whatsoever — trust is 100% delegated to whatever installed the server.

## 2. Supply-chain attack patterns and proven mitigations

### The pattern that recurs everywhere: credential leakage → automated republish, not initial malicious upload

Across VS Code, Open VSX, npm, and PyPI, the *dominant* 2025–2026 attack pattern was **not** a human manually vetting and approving a first-time malicious submission. It was an attacker stealing a *publish credential* from a legitimate, already-trusted publisher and using it to push a malicious *update* to an existing, previously-approved package or extension — instantly inheriting that publisher's entire install base and trust signals (verified badge, download count, age). This is structurally the same attack across ecosystems built on completely different tech stacks, which strongly suggests it's a property of the trust model (identity-bound publish tokens with no per-publish human gate), not a code-specific vulnerability. ✅ (corroborated across VS Code/Open VSX, npm, and Chrome extension incidents independently)

**Concrete, well-corroborated incidents illustrating this:**

- **May 18–19, 2026 — GitHub internal breach via poisoned Nx Console VS Code extension.** ✅ Confirmed directly: GitHub confirmed ~3,800 internal repositories exfiltrated after an employee installed a trojanized version of the popular Nx Console extension; GitHub linked the compromise to a related TanStack npm supply-chain attack; attacker group "TeamPCP" advertised the stolen source for sale. This happened on the *official, actively-scanned* Marketplace. Multiple outlets (BleepingComputer, VentureBeat, Help Net Security, The Hacker News, Security Affairs) corroborate the core facts; the specific "18-minute exposure window" detail was reported by some outlets but not independently reconfirmed in our direct-fetch verification — treat that specific figure as ⚠️ pending re-check.
- **October 2025 — Open VSX "GlassWorm" token leak.** ✅ Eclipse Foundation's own incident report (Oct 27, 2025) confirms: leaked publish tokens (found in public repos via a separate Wiz secret-scanning finding of 550+ exposed secrets across 500+ VS Code/Open VSX extensions) were used to push malicious extension updates. Root cause was developer token hygiene, not registry compromise. Remediation: token revocation, new scannable token-prefix format, shorter token lifetimes, and — notably — automated pre-publish security scanning was **only added as a result of this incident**, meaning it didn't exist before.
- **September 2025 — "Shai-Hulud" self-propagating npm worm.** ✅ CISA issued an alert (Sept 23, 2025). Malicious postinstall scripts harvested secrets/tokens from the host and public GitHub repos using a bundled TruffleHog binary, then auto-republished trojanized versions of every package the stolen token could reach — 194 packages / 582 versions affected, including CrowdStrike-owned packages. This is the same "stolen publish credential → automated re-publish" pattern as the VS Code/Open VSX incidents, but weaponized to *self-propagate*.
- **December 2024–2025 — Chrome extension developer spearphishing.** ✅ Attackers phished extension developers with fake "policy violation" emails to steal OAuth publishing credentials, then pushed malicious updates; Secure Annex tracked 25 compromised extensions with ~2.29M combined users. Structurally identical to the above two.
- **March–June 2026 — Axios, node-ipc, Mastra AI npm compromises.** Axios (100M+ weekly downloads) was compromised via a stolen npm token that bypassed GitHub Actions OIDC trusted-publisher protections entirely (attacker published directly, leaving no trace in the source repo) — a reminder that provenance/OIDC protections only help if publishing is *actually* routed through the protected CI pipeline; a stolen classic token sidesteps them completely. ✅ (5 independent sources on Axios). node-ipc and Mastra AI are ⚠️ single-sourced in this pass but follow the identical pattern.

### Proven mitigations, ranked by how directly they close the "stolen credential → silent republish" gap

1. **Mandatory 2FA on publish, with growing 2FA-gated (human-approved) publishing.** npm's 2FA mandate started with top-100 maintainers (2022), expanded to all high-impact packages, and by May 2026 added **"2FA-gated publishing"** requiring a live human 2FA challenge to approve a package *before* it reaches the public registry, plus 7-day token lifetime caps. ✅ This is the single most directly-applicable mitigation to the pattern above, because it converts "steal a long-lived token" into "steal a token AND compromise a live 2FA challenge," which is a categorically harder attack.
2. **Provenance/attestation (Sigstore-backed OIDC), but adoption is low and it doesn't stop a stolen-token direct-publish.** npm/PyPI provenance cryptographically ties a package to the CI build that produced it (short-lived Sigstore-issued cert, discarded key, public transparency log). GA since Oct 2023 (npm) / trusted publishing GA July 2025. However: only ~12% of top-50 npm packages use provenance as of an April 2026 audit ⚠️ (single-source blog, not an official stat), and the Axios 2026 attack proved that a stolen classic token bypasses provenance entirely because the attacker never goes through the CI pipeline the provenance system is watching. **Provenance is a positive-trust signal, not an access-control gate** — this distinction matters for Sonik.
3. **Reactive malware scanning at publish time**, added by Open VSX only after being breached, and expanded at npm/PyPI/VS Code Marketplace through 2025. Catches known-signature and pattern-based malware; caught the "prettier-vscode-plus" impersonation extension within 4 hours on VS Code Marketplace (Nov 2025) — evidence the approach *can* work fast, but only after-the-fact, and it did not stop the Nx Console breach that followed six months later.
4. **Quarantine mechanisms** (PyPI's Project Quarantine, Dec 2024) let registry admins freeze a suspicious package (hidden from index, uninstallable, frozen from owner edits) while investigating — a useful "kill switch" pattern that doesn't require proving malice first, just reasonable suspicion.
5. **Lockfile hash verification** protects against tarball tampering in transit but explicitly does **not** protect against a legitimately-signed malicious update — exactly the vector used in every incident above. Worth stating precisely because it's commonly over-trusted as a supply-chain defense.

### Authoritative framework guidance (2025–2026, directly applicable to Sonik as an AI agent platform)

- **OWASP Top 10:2025** added a new, elevated category — **A03:2025 "Software Supply Chain Failures"** — explicitly citing Shai-Hulud as the motivating incident, and recommending SBOM/SCA tooling and a rule that "no single person can push code to production without independent review." ✅
- **OWASP MCP Top 10 (2025)**, a new project specific to MCP, includes **"MCP04:2025 – Software Supply Chain Attacks & Dependency Tampering"** — directly on point for anything Sonik builds on an MCP-like tool/plugin pattern. ✅ Recommend Sonik's team read this document directly.
- **NIST SSDF v1.2** (draft, Dec 17, 2025) strengthens delivery-safety practices (PS.4: explicit rollout/rollback protections) and adds config-testing requirements (RV.1.2). **NIST SP 800-218A** extends SSDF specifically to generative AI / dual-use foundation models — worth pulling directly given Sonik's domain.
- **CISA/NSA Enduring Security Framework** guidance on supply-chain defense (customer/supplier/developer-facing) remains the baseline cross-industry reference.

## 3. Agent-specific risks: prompt injection and tool poisoning via installed content

This is the newest and least mature part of the landscape, and it maps almost one-to-one onto Sonik's stated gaps (unregistered free-string capability IDs, no install-time human gate, off/ask/allow not yet runtime-enforced).

### Tool Poisoning Attacks (TPA) — the foundational finding

Invariant Labs coined and disclosed this April 1, 2025 (directly verified, primary source): **a malicious MCP tool's description can contain instructions that are invisible in the UI a user sees, but fully visible to the model, which follows them.** ✅ The canonical example: a tool description contains hidden `<IMPORTANT>` tags instructing the model to read `~/.ssh/id_rsa` and a Cursor config file, then exfiltrate their contents disguised as an innocuous parameter, while the user's UI shows only a simplified tool summary. A follow-up week-later post demonstrated the same technique exfiltrating full WhatsApp chat histories. This is not a theoretical concern — it's a structural property of how tool descriptions get merged into an LLM's context: **the model cannot distinguish "genuine tool documentation" from "adversarial instructions embedded in that documentation," and the protocol has no built-in mechanism to detect or flag this.** ✅

Related named attack classes from the same and adjacent research (all corroborated by 2+ independent sources — OWASP MCP Cheat Sheet, ReversingLabs, PolicyLayer, academic ETDI paper):
- **Rug pull**: a tool passes review, then its description or behavior is silently rewritten *after* the user has already approved it — exploiting the gap between one-time approval and every subsequent invocation. There is a named CVE for this pattern ("MCPoison," CVE-2025-54136).
- **Shadowing**: a malicious server's tool description manipulates how the agent uses a *different*, legitimate, already-trusted tool loaded in the same context — because all tool descriptions share one context window and the model can't segregate trust boundaries between them.
- **Confused deputy / token redirection**: covered explicitly in the MCP spec's own Security Considerations section (directly verified) — the spec acknowledges "mix-up and confused deputy attacks" as a known risk class and mandates resource-indicator binding (RFC 8707) and audience validation specifically to mitigate it. This is a case where the spec *has* caught up with a known risk, in contrast to tool-description integrity, which it has not addressed at all.

**Benchmark finding, not yet a confirmed real-world breach as of the underlying study**: tool-poisoning attacks achieved up to 84.2% success rates in controlled lab testing specifically when the target agent has auto-approval enabled ⚠️ (single-study statistic, mid-2025 vintage — likely stale given the incident list below).

### Real-world MCP incidents/CVEs since (severity escalating through 2025–2026)

⚠️ Several of the following trace to a single aggregator (Authzed's incident timeline) and should be corroborated against primary disclosures before being cited externally, but the pattern across them is consistent and the CVE-level findings are independently listed in the Vulnerable MCP Project database:
- CVE-2025-6514 (July 2025): OS command injection in the popular `mcp-remote` OAuth proxy (437K+ downloads), CVSS 9.6 — a malicious/untrusted remote MCP server could achieve RCE on the connecting client.
- CVE-2025-53109/53110 (Aug 2025): sandbox escape and symlink bypass in Anthropic's own official filesystem MCP server.
- CVE-2026-0755 (Jan 2026, CVSS 9.8) and CVE-2026-23744 (Feb 2026, CVSS 9.8): unsanitized command execution and no-auth-by-default RCE in community MCP tooling.
- May 2025 GitHub MCP incident: a malicious public GitHub issue hijacked a coding agent into exfiltrating private repo contents through an over-scoped PAT passed via the GitHub MCP server. ⚠️
- Feb 2026: a trojanized clone of the Oura MCP project distributed StealC infostealer malware via public package registries. ⚠️

### Government recognition of the threat model

**May 2026: NSA's AI Security Center published dedicated guidance** on MCP specifically, describing the protocol's design as "flexible and underspecified" and explicitly recommending least-privilege per-tool tokens, signed-provenance dynamic discovery, and treating registries as hardened API gateways. ✅ CISA, NSA, UK NCSC, Australian Signals Directorate, and Canada's CCCS jointly released a companion agentic-AI security guidance document the same period (May 1, 2026) — the first cross-government guidance specifically targeting this class of risk. This is a strong signal that "installed executable AI content = distinct new risk category requiring distinct controls" is now consensus among Western security agencies, not a niche researcher concern.

### Precedent from earlier agent-plugin ecosystems

ChatGPT plugins (2023, since deprecated) had a documented OAuth flaw that could let an attacker install a malicious plugin onto a victim's account without consent (Salt Labs, June 2023) — an early instance of the exact "chat content secretly authorizing an install/action" failure mode that Sonik's "chat text is never authorization" principle is explicitly designed to prevent. ✅ Worth noting as validation that Sonik's baseline principle already avoids a known, previously-exploited failure class.

## 4. Recommendations mapped onto Sonik's model — ranked by risk

Sonik's stated baseline is sound in its core design principle (chat is never authorization; renderer specs are execution-inert; mutating commands require host-signed `approvedCommandIds`). The risk is concentrated almost entirely in the **gaps**, and the industry evidence above lets us rank them by how directly they map to incidents that have already happened to other people.

**#1 — No human-in-the-loop gate at install time is the highest-risk gap, and it is exactly the gap that caused the worst incidents surveyed.** Every headline incident in this report — Nx Console/GitHub, GlassWorm/Open VSX, Shai-Hulud, the Chrome developer-phishing wave — succeeded specifically at the *install/update* moment, not at runtime. Sonik currently has no equivalent of even VS Code's publisher-trust dialog. **Recommendation**: before any marketplace-sourced package/skill/connector can register a capability, require an explicit, human-visible install-time screen naming (a) the publisher, (b) every capability/command family being requested, and (c) whether any requested command is mutating. This does not need to be as elaborate as Google's CASA tiers — even Chrome's "read and change your data on this site"-style plain-language disclosure closes most of the gap, because the *absence* of any screen is what every incident above exploited.

**#2 — Unregistered free-string capability IDs is the second-highest risk, because it defeats every other control you might add later.** A permission system (off/ask/allow) is only as strong as the identifier it's keyed on. Free-string capability IDs mean two different packages can claim the same capability name with different actual behavior, or a single package can rename a capability across versions to dodge a previously-set "off." This is structurally the same weakness that let VS Code's "verified publisher" badge get exploited (OX Security, 2025: badge persisted even after malicious functionality was added, because the check wasn't re-evaluated against changed behavior) and the same weakness Anthropic's own engineering post flags for remote MCP tools ("can change behavior at any point after you've approved them"). **Recommendation**: stand up a registered capability-ID namespace (even a simple centrally-issued enum, versioned) before scaling the marketplace — this is cheap now and extremely expensive to retrofit once third-party packages exist with squatted or ambiguous IDs.

**#3 — off/ask/allow not yet enforced at runtime is high risk but time-bounded — fix before, not after, marketplace scale.** Anthropic's own data is the most directly relevant evidence here: when Claude Code required approval on every action, users approved ~93% of prompts ("approval fatigue"), meaning a human-in-the-loop gate that fires *too often* degrades to a rubber stamp and provides a false sense of safety. The fix Anthropic converged on was not "prompt more," it was OS-level sandboxing to shrink blast radius (84% reduction in prompts needed) plus a probabilistic pre-execution classifier — i.e., invest in **reducing how often a prompt is needed at all**, not just building the prompt UI. **Recommendation**: when Sonik implements runtime enforcement of off/ask/allow, budget for approval-fatigue from day one — pair it with sandboxing/blast-radius containment for "ask"-tier commands rather than relying on the prompt alone, and log approval-rate metrics to catch rubber-stamping early.

**#4 — Uninstall/revocation being unspecified is a real but currently lower-probability risk, based on how it manifests elsewhere.** The clearest 2026 evidence (Vercel breach, Klue breach) shows that dangling/zombie access from a *third party's* stored credentials, not the victim's own uninstall state, was the actual failure mode — i.e., the risk isn't "user forgets to uninstall," it's "a package Sonik trusted got compromised and had standing tokens that outlived any reasonable revocation window." Shopify's own documented pattern (token invalidated before the uninstall webhook fires, forcing reliance on a delayed GDPR-style redact webhook) shows even mature platforms haven't solved instant, complete revocation. **Recommendation**: design revocation as "stop trusting this credential going forward" (fast, achievable) rather than promising "and all downstream copies are gone" (not achievable) — and separately maintain a centralized inventory of every standing grant per package, since "no one has a full inventory of active integrations" was the root cause named across multiple 2026 breach post-mortems.

**#5 — No template/content moderation is a real gap but is the least differentiated from problems the whole industry is still failing to solve; don't over-invest relative to #1–#3.** Prompt injection via installed content (the WhatsApp/SSH-key TPA examples, EchoLeak's zero-click Microsoft 365 Copilot exploit) is explicitly acknowledged by Anthropic, OpenAI, and Google DeepMind as not fully solvable within current LLM architectures ⚠️ (this vendor-consensus framing is itself only secondhand-sourced in our research pass and should be checked against primary vendor statements before being asserted externally). Sonik's execution-inert renderer spec design (no tool_call/endpoint/commit payloads in JSON) is already a stronger structural defense than most of what we found in the industry survey — it removes the injection *target*, rather than trying to detect injected *content*. **Recommendation**: keep leaning on architectural inertness rather than chasing content-moderation detection (a losing arms race per current industry consensus); do add basic typosquat/name-collision checks at publish time (Socket.dev's Levenshtein-distance approach against existing capability names is cheap and catches a meaningful slice of real attacks).

## So what — five takeaways

- **Every mainstream marketplace we examined has been breached through its own trust mechanism within the last 12–18 months**, usually via a stolen publish credential rather than a cleverly-disguised first-time malicious upload — Sonik should assume its own future publish/update path will be targeted the same way, not just the initial submission.
- **A "verified" or "trusted" badge is a snapshot, not a standing guarantee** — both VS Code (OX Security research) and Anthropic's own engineering team (on remote MCP tools) independently arrived at the same conclusion: behavior can change after approval, so trust needs re-attestation on update/behavior-change, not a permanent stamp.
- **Per-action approval prompts fail from fatigue, not from bad UI** — Anthropic's measured 93% blanket-approval rate is direct evidence that "ask" alone is not a control; it needs to be paired with blast-radius reduction (sandboxing, scoped tokens) or it becomes theater.
- **MCP's own spec has no answer for tool-description integrity** — it solved OAuth/confused-deputy problems in 2025 revisions but explicitly leaves authorization optional (and drops it entirely for STDIO transport), meaning most local, developer-run MCP servers today have zero protocol-level trust boundary; Sonik cannot assume MCP-level protections exist and must build its own.
- **Sonik's core doctrine (chat is never authorization, execution-inert renderer specs) already avoids the single most-exploited historical failure mode** (OAuth-consent-style "chat/click secretly grants access," as seen in the 2017 Google Docs worm and the 2023 ChatGPT plugin OAuth flaw) — the real exposure is concentrated in the five install/runtime gaps above, and #1 (no install-time human gate) and #2 (unregistered capability IDs) should be closed before onboarding third-party marketplace packages at any scale.
