TOOL CREATION · SYSTEM EXPLAINER · 2026-07-07

One registry. Many transports. One trusted execution boundary.

Sonik Agent UI should treat every agent-callable capability as a registry-backed command contract: discovered compactly, learned just-in-time, executed through the correct adapter, approved only by trusted host/session state, and reported back as a typed receipt.

The one mental model

Tools are not arbitrary functions. They are command descriptors with provenance, policy, scope, transport, receipts, and telemetry.

1. Discover

Use compact indexes: searchCommandCatalog, searchSkillCatalog, or marketplace/template search. No full schema flood.

2. Learn

Load details only after selection: schema, examples, policy, transport, auth, output, and relevant surfaces.

3. Prepare

Validate input, page context, active artifact state, required host session, and tool permission mode.

4. Execute

Reads use execute. Writes use commit and derive approval from trusted host/session state.

5. Receipt

Return command id, policy decision, trace/provider, safe output, next action, and telemetry.

Tool families and ownership

Families are registry-backed taxonomy. Core owns neutral families; hosts provide domain vocabulary.

Family / source What it represents Execution boundary Current posture
local-ui Artifact, document, canvas, layout, local state, future host-action requests. Agent UI/controller. Durable mutation still needs commit/approval if stateful. EXISTS host actions pending
orpc / openapi Typed app/business operations such as booking, campaigns, guests, reservations. Generated descriptors remain shadow until mounted by a host runtime adapter. EXISTS generated
mcp External/desktop tool projection with preserved provenance. Search/learn/execute/commit/resource patterns; not first-party trusted by default. surface-dependent
sandbox / cli Environment, shell, filesystem, code, or external process work. Deny by default; trusted host enablement and resource limits required. restricted
json-render artifact Schema-backed UI, forms, dashboards, questions, previews. Renderer emits typed answer/state events only; controller translates to commands. EXISTS needs richer action state
skill registry Markdown process instructions and workflow recipes. Skills teach the agent; they do not grant execution authority. EXISTS

Execution lifecycle and trust split

The same registry model works across reads, writes, local UI, artifacts, and external commands because trust is separated from model text.

What is different between ORPC, CLI, registry, and artifacts?

The apparent overlap is resolved by source ownership and execution authority.

Surface Primary job Can mutate business state? Correct usage Incorrect usage
Command registry Canonical index of agent-callable descriptors. Only through mounted runtime + policy. Search compactly, learn exact schema, execute/commit with receipt. Hardcode all tools into prompt context.
ORPC/OpenAPI Typed business contracts generated from host service truth. Yes, when mounted and approved. Generate descriptors; keep shadow until adapter is live. Handwrite 72 endpoints from memory or call private service internals.
CLI/MCP projection Alternate interface over the same search/learn/execute/commit semantics. Maybe, but never by bypassing policy. Preserve provenance, approval, receipts, and resource limits. Create separate business logic that drifts from ORPC/registry.
Artifact creation Create schema-backed UI and document/canvas state. Artifact persistence yes; business mutation no. Use forms/questions/previews to collect typed user input. Embed executable endpoint strings or commitCommand payloads in JSON specs.
Host action channel Safe UI control: highlight, focus, open panel, request approval card. No business mutation directly. Semantic target IDs + host validation + receipts. Let the iframe click arbitrary DOM or treat tour steps as command approvals.
Skills Teach process: what to search, learn, ask, preview, and avoid. No. Use with command catalog and artifact tools. Assume a skill grants a command or approval.

Where Driver.js-style tours fit

Driver.js is a behavior donor. The platform primitive is a host action channel, implemented as local-ui/host-adapter command descriptors.

Define contract types

AgentHostActionRequest, AgentHostActionResult, target registry, policy mode, and receipt shape.

Expose as registry commands

Examples: host.highlightTarget, host.openCanvas, host.requestApproval.

Host resolves semantic targets

The agent requests booking-context.schedule; the host maps it to the current DOM/component safely.

Return receipts

The iframe must receive ok, target_not_found, approval_required, or host_unavailable. No silent pretend-success.

Contradiction audit

This audit checks the logic against Sonik tool creation, Sonik accessibility, component design, and current repo behavior.

PASS

One command doctrine

ORPC/OpenAPI, local UI, MCP, sandbox, artifacts, and future host actions can all sit behind one descriptor/search/learn/execute/commit/receipt model.

PASS

Renderer authority boundary

JSON-render components collect typed input and emit state/action events. They must not embed business endpoints or executable command payloads.

PASS

Model cannot approve

Approval must bind command id, target, input hash, actor/session/org, expiry, and idempotency. Chat text is only intent.

WATCH

Host action inventory missing

We have business command inventory, but still need a first-class inventory of host UI targets and allowed host actions.

WATCH

Artifact actions need richer receipts

QuestionCard and ActionRail exist, but the user-facing approval/save/submit model needs clearer state, disabled reasons, and action receipts.

DO NOT DO

No vertical/event drift

Restaurants, tee sheets, gyms, hotels, and resources should use venue_schedule or resource, not fake daily Events.

Inventory and next steps

The next implementation should start with contracts and tests, then mount runtime adapters. Do not start by styling a tour overlay.

Needed inventory Where it should live Why it matters First test
Host action contract packages/tool-contracts/src/host-actions.ts Gives agent UI and host apps one typed request/result protocol. Unknown action fails closed with typed receipt.
Semantic UI target registry Host adapter + page context envelope Avoids brittle CSS selectors and lets hosts choose their own vocabulary. Unknown target returns target_not_found.
Host action command descriptors Command catalog adapter, family ui Keeps tours/page-control inside existing tool creation doctrine. Surface-eager on matching host context only.
Approval card action Host UI + Agent UI receipt view Lets agents request approval UX without approving themselves. Chat "approve" alone cannot execute command.
Driver-style tour adapter Later UI layer behind host actions Turns behavior parity into polish without making Driver.js the architecture. Tour step succeeds only when target registry resolves.

Final answer

The current logic is directionally consistent if we keep the vocabulary strict.

What is already right
  • Search/learn avoids flooding context.
  • ORPC/OpenAPI commands are generated and mounted through adapters.
  • Writes go through commit and trusted approval, not model booleans.
  • Artifacts are schema-backed input/UI surfaces, not command authorities.
  • Skills guide process without granting execution rights.
What needs to be added next
  • Host action contract and receipt schema.
  • Semantic UI target inventory for Booking and Amplify.
  • Local-ui command descriptors for host page actions.
  • Tests for blocked/ask/allow/require policy behavior.
  • Approval card UX that is visibly separate from chat text.
